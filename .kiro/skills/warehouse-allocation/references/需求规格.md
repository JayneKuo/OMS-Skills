# 寻仓引擎

寻源漏斗是分仓决策的核心过滤链路，按 P0 → P1 → P2 → P3 四级依次执行，不可跳级或并行。每一级过滤后，如果候选集为空，立即触发对应的异常兜底逻辑，不继续往下走。

## 7.1.1 P0：硬约束过滤（不可违反）

P0 阶段执行 8 条硬约束，任何一条不满足即淘汰该候选仓：

| 编号 | 硬约束 | 说明 |
|------|--------|------|
| P0-1 | 仓状态可用 | 仓库处于"正常营业"状态，未停用、未冻结、未在盘点封仓期 |
| P0-2 | 商家已开通该仓 | 商家与该仓存在有效的合作关系且未过期 |
| P0-3 | SKU 在该仓有可用库存 | `available_qty ≥ order_qty`，可用库存 = 实物库存 − 已预占 − 已锁定 − 安全库存 |
| P0-4 | 配送范围覆盖 | 收货地址在该仓的配送范围内（基于行政区划或围栏） |
| P0-5 | 承运商可达 | 至少存在一个承运商可以从该仓配送到收货地址 |
| P0-6 | 温区匹配 | 订单商品的温区要求（常温/冷藏/冷冻）与仓库支持的温区一致 |
| P0-7 | 合规限制通过 | 商品不在该仓所在地区的禁售/限售清单中（如跨境、特殊品类） |
| P0-8 | 物理可发 | 商品的物理属性（超长/超重/危险品）在该仓的发货能力范围内 |

P0 阶段淘汰的候选仓必须记录淘汰原因（用于白盒展示），格式为四元组：`What → Why → How → ETA`。

## 7.1.2 P1：业务条件过滤

P1 阶段在 P0 通过的候选仓中，进一步检查 3 个业务条件：

| 编号 | 条件 | 说明 |
|------|------|------|
| P1-1 | 截单时间未过 | 当前时间 < 该仓当日截单时间（cut-off time），否则最早发货日顺延至下一工作日 |
| P1-2 | 仓容量未超限 | 该仓当日已接单量 < 仓日处理能力上限 × 容量系数（capacity_ratio） |
| P1-3 | 承运商时效满足 SLA | 至少存在一个承运商的预估送达时间 ≤ 客户要求的最晚送达时间 |

### SLA 裁决优先级

当多个仓都满足 P1 条件时，按以下优先级裁决：

| 优先级 | 规则 | 说明 |
|--------|------|------|
| 1 | 客户指定仓 | 如果客户/商家明确指定了发货仓，且该仓通过 P0+P1，直接选定 |
| 2 | 平台强制规则 | 平台级规则（如大促期间指定仓、区域管控）优先于算法计算 |
| 3 | 商家优先仓 | 商家配置的优先发货仓，在同等条件下优先选择 |
| 4 | 算法最优解 | 以上均无时，进入 P2 评分排序 |

## 7.1.3 P2：多维评分排序

P2 阶段对通过 P0+P1 的候选仓进行多维评分，计算综合得分：

```
Score_warehouse = w_cost × S_cost + w_eta × S_eta + w_capacity × S_capacity + w_stability × S_stability
```

各维度说明：

| 维度 | 权重字段 | 说明 |
|------|----------|------|
| S_cost | w_cost | 综合成本得分（运费 + 仓操费 + 拆单惩罚），越低越好，归一化后取反 |
| S_eta | w_eta | 预估送达时效得分，越快越好，归一化后取反 |
| S_capacity | w_capacity | 仓剩余容量得分，剩余容量越多越好 |
| S_stability | w_stability | 仓稳定性得分，基于近期异常率、准点率等 |

权重默认值：`w_cost = 0.40, w_eta = 0.30, w_capacity = 0.15, w_stability = 0.15`，支持按商家/品类/大促模式动态调整。

所有子分数均归一化到 [0, 1] 区间，归一化方式为 min-max 归一化：

```
S_normalized = (value - min) / (max - min)
```

## 7.1.4 P3：稳定性兜底规则

P3 阶段在 P2 排序结果上叠加 4 条稳定性规则，防止频繁切仓和仓网震荡：

| 编号 | 规则 | 说明 |
|------|------|------|
| P3-1 | 同商家同 SKU 粘性 | 同一商家的同一 SKU，在过去 N 小时内如果持续分配到某仓，优先继续分配到该仓，除非该仓评分低于阈值 |
| P3-2 | 切仓冷却期 | 某仓被切走后，在冷却期（默认 30 分钟）内不会被重新切回，防止乒乓效应 |
| P3-3 | 最大切仓频率 | 同一商家同一 SKU 在 1 小时内最多切仓 N 次（默认 N=2） |
| P3-4 | 容量均衡保护 | 当 Top1 仓和 Top2 仓评分差距 < 阈值（默认 5%）时，优先选择当前容量利用率更低的仓 |

P3 规则仅调整排序，不淘汰候选仓。如果 P3 调整后的 Top1 仓与 P2 的 Top1 仓不同，需记录切换原因。

## 7.2 候选仓生成

### 7.2.1 入围条件

候选仓必须同时满足以下 5 个入围条件，才能进入评分阶段：

| 编号 | 条件 | 说明 |
|------|------|------|
| 1 | 通过 P0 硬约束 | 该仓已通过寻源漏斗 P0 阶段全部 8 条硬约束 |
| 2 | 通过 P1 业务条件 | 该仓已通过寻源漏斗 P1 阶段全部 3 条业务条件 |
| 3 | SKU 覆盖率 ≥ 阈值 | 该仓可满足的 SKU 数量 / 订单总 SKU 数量 ≥ sku_coverage_threshold（默认 100%，拆单场景可降低） |
| 4 | 库存满足率 ≥ 阈值 | 该仓可满足的商品件数 / 订单总件数 ≥ qty_coverage_threshold（默认 100%） |
| 5 | 预估 ETA ≤ SLA 上限 | 该仓的预估送达时间不超过客户要求的最晚送达时间 |

### 7.2.2 局部评分

每个候选仓的局部评分公式：

```
Score_local(w) = w_cost × S_cost(w) + w_eta × S_eta(w) + w_cap × S_cap(w) + w_stab × S_stab(w)
```

各维度权重默认值：

| 维度 | 权重 | 默认值 | 说明 |
|------|------|--------|------|
| w_cost | 成本权重 | 0.40 | 综合成本越低越好 |
| w_eta | 时效权重 | 0.30 | 送达越快越好 |
| w_cap | 容量权重 | 0.15 | 剩余容量越多越好 |
| w_stab | 稳定性权重 | 0.15 | 近期异常率越低越好 |

### 7.2.3 归一化口径

所有子分数归一化到 [0, 1] 区间，使用以下 5 个公式：

```
S_cost(w) = 1 − (Cost(w) − Cost_min) / (Cost_max − Cost_min)
```

```
S_eta(w) = 1 − (ETA(w) − ETA_min) / (ETA_max − ETA_min)
```

```
S_cap(w) = (RemainCap(w) − RemainCap_min) / (RemainCap_max − RemainCap_min)
```

```
S_stab(w) = (OnTimeRate(w) − OnTimeRate_min) / (OnTimeRate_max − OnTimeRate_min)
```

```
当 max = min 时，S = 1.0（所有候选仓该维度相同，不产生区分度）
```

3 个封顶规则（Caps）：

| 编号 | 封顶规则 | 说明 |
|------|----------|------|
| Cap-1 | Cost 封顶 | 当 Cost(w) > Cost_cap 时，S_cost(w) = 0，该仓成本维度直接归零 |
| Cap-2 | ETA 封顶 | 当 ETA(w) > ETA_cap 时，S_eta(w) = 0，该仓时效维度直接归零 |
| Cap-3 | 容量封顶 | 当 RemainCap(w) < Cap_floor 时，S_cap(w) = 0，该仓容量维度直接归零 |

### 7.2.4 权重归属规则

权重的确定遵循以下 4 条规则，按优先级从高到低：

| 优先级 | 规则 | 说明 |
|--------|------|------|
| 1 | 订单级 Override | 如果订单携带了自定义权重（如 API 传入），使用订单级权重 |
| 2 | 商家级配置 | 商家在后台配置的权重偏好（如某商家更看重时效） |
| 3 | 品类级配置 | 按商品品类配置的权重（如生鲜品类时效权重更高） |
| 4 | 系统默认值 | 使用系统默认权重：w_cost=0.40, w_eta=0.30, w_cap=0.15, w_stab=0.15 |

### 7.2.5 候选仓裁剪规则

为控制后续方案枚举的计算量，对候选仓进行裁剪：

| 编号 | 规则 | 说明 |
|------|------|------|
| 1 | Top-K 截断 | 只保留评分最高的 K 个候选仓（默认 K=10） |
| 2 | 最低分淘汰 | Score_local < score_floor（默认 0.2）的候选仓直接淘汰 |
| 3 | 同质仓合并 | 如果两个仓的评分差距 < merge_threshold（默认 2%）且属于同一仓群，只保留评分更高的一个 |

### 7.2.6 典型案例

**案例：某订单包含 3 个 SKU，共 5 件商品**

- 仓 A：3 个 SKU 全部有货，库存充足，ETA 2 天，成本 15 元 → 入围
- 仓 B：仅 2 个 SKU 有货（覆盖率 66.7%），低于阈值 → 单仓不入围，但可作为拆单候选
- 仓 C：3 个 SKU 全部有货，但 ETA 5 天，超过 SLA 上限 3 天 → 不入围
- 仓 D：3 个 SKU 全部有货，ETA 2.5 天，成本 12 元 → 入围

最终候选仓：A、D 进入方案枚举阶段。

## 7.3 方案枚举

### 7.3.1 允许方案类型

系统支持以下 5 种方案类型：

| 编号 | 方案类型 | 说明 |
|------|----------|------|
| 1 | 单仓直发 | 所有商品从同一个仓发出，一个包裹送达 |
| 2 | 单仓多包 | 所有商品从同一个仓发出，但因装箱限制拆为多个包裹 |
| 3 | 多仓拆发 | 商品分别从不同仓发出，客户收到多个包裹 |
| 4 | 调拨合发 | 将缺货 SKU 从其他仓调拨到主仓，合并后统一发出 |
| 5 | 部分发货 + 缺货登记 | 有货部分先发，缺货部分登记待补发（需商家/客户确认） |

### 7.3.2 枚举顺序

方案枚举按以下 6 步顺序执行：

| 步骤 | 动作 | 说明 |
|------|------|------|
| 1 | 尝试单仓直发 | 遍历所有候选仓，检查是否有仓可以单独满足全部 SKU 和数量 |
| 2 | 尝试单仓多包 | 对单仓直发可行的仓，执行装箱计算，如果需要拆为多包则生成单仓多包方案 |
| 3 | 尝试多仓拆发 | 如果没有单仓可满足，枚举 2 仓组合、3 仓组合...直到覆盖全部 SKU |
| 4 | 尝试调拨合发 | 对每个候选主仓，检查缺货 SKU 是否可以从其他仓调拨过来 |
| 5 | 尝试部分发货 | 如果以上方案都无法覆盖全部 SKU，生成部分发货方案 |
| 6 | 无解兜底 | 所有方案均不可行，进入异常处理流程 |

### 7.3.3 剪枝条件

为控制枚举复杂度，设置以下 7 条剪枝条件：

| 编号 | 条件 | 说明 |
|------|------|------|
| 1 | 最大拆单仓数 | 单个订单最多拆分到 N 个仓（默认 N=3），超过则剪枝 |
| 2 | 成本上界剪枝 | 如果当前部分方案的成本已超过已知最优方案的成本，剪枝 |
| 3 | ETA 上界剪枝 | 如果当前部分方案的最大 ETA 已超过 SLA 上限，剪枝 |
| 4 | 重复组合剪枝 | 仓的组合不区分顺序，{A,B} 和 {B,A} 视为同一方案 |
| 5 | 库存不足剪枝 | 如果剩余未分配 SKU 在所有剩余候选仓中都无库存，提前终止 |
| 6 | 容量溢出剪枝 | 如果某仓已接近容量上限，不再向该仓分配更多 SKU |
| 7 | 调拨距离剪枝 | 调拨距离超过 max_transfer_distance（默认 500km）的调拨方案直接剪枝 |

### 7.3.4 超时降级策略

方案枚举设置总超时时间（默认 2000ms），超时后按以下 5 级降级：

| 级别 | 触发条件 | 降级动作 |
|------|----------|----------|
| L1 | 枚举时间 > 500ms | 停止 3 仓及以上组合枚举，只保留 1 仓和 2 仓方案 |
| L2 | 枚举时间 > 1000ms | 停止调拨合发方案枚举 |
| L3 | 枚举时间 > 1500ms | 停止多仓拆发枚举，只保留已找到的单仓方案 |
| L4 | 枚举时间 > 2000ms | 直接选择当前已找到的最优方案，停止枚举 |
| L5 | 枚举时间 > 3000ms | 触发熔断，使用兜底规则（最近仓 + 有货）直接出结果 |

### 7.3.5 调拨合发准入规则

调拨合发方案必须同时满足以下 5 个条件：

| 编号 | 条件 | 说明 |
|------|------|------|
| 1 | 主仓覆盖率 ≥ 阈值 | 主仓已有 SKU 覆盖率 ≥ transfer_main_coverage（默认 60%），避免大量调拨 |
| 2 | 调拨仓数 ≤ 上限 | 需要调拨的源仓数量 ≤ max_transfer_sources（默认 2） |
| 3 | 调拨时效可接受 | 调拨到达主仓的时间 + 主仓处理时间 + 配送时间 ≤ SLA 上限 |
| 4 | 调拨成本可接受 | 调拨运费 + 主仓发货运费 < 多仓拆发总运费 × transfer_cost_ratio（默认 0.9） |
| 5 | 调拨通道可用 | 主仓与源仓之间存在可用的调拨通道（物流线路已开通且未暂停） |

## C. 库存口径说明

PRD 定义的可用库存公式：
```
available_qty = physical_qty - reserved_qty - locked_qty - safety_stock
```

MVP 实际使用：
- inventory/list API 返回 `onHandQty`（实物库存），不返回 `availableQty`
- MVP 用 `onHandQty` 近似 `available_qty`
- 这意味着可能高估可用库存（因为没有扣除预占和锁定）
- 降级标记：`inventory_degraded=true`，置信度降为 medium

## D. 距离估算算法

MVP 使用美国州级中心点坐标查表计算直线距离：

- 数据源：美国 50 州 + DC 的中心点经纬度（硬编码）
- 距离公式：Haversine 公式计算两点间大圆距离
- 距离 → 成本映射：`estimated_cost = base_cost + distance_km × cost_per_km`（base_cost=$5, cost_per_km=$0.02）
- 距离 → 时效映射：`estimated_days = max(1, distance_km / 500)`
- 同州：距离=0，成本=base_cost，时效=1 天
- 无州信息：距离=最大值（5000km），成本/时效得分=0

局限性：
- 州级精度粗糙，同州内不同城市可能差距很大
- 直线距离 ≠ 实际运输距离
- 成本映射是线性估算，实际运费有阶梯/区域定价

## E. 数据降级策略

| 缺失数据 | 降级策略 | 置信度影响 |
|----------|----------|-----------|
| availableQty 不可用 | 用 onHandQty 近似 | → medium |
| 仓库无容量数据 | 容量维度得分=1.0（不区分） | → medium |
| 仓库无州/坐标信息 | 距离=最大值，成本/时效得分=0 | → medium |
| 库存数据为空 | 该仓排除 | 不影响 |
| 订单无收货地址 | 阻断，返回错误 | — |
| 订单无商品行 | 阻断，返回错误 | — |
| 温区数据缺失 | 跳过温区检查 | → medium |

## E-2. 外部数据不可用的降级策略

以下数据在 PRD 中被引用，但当前 OMS/WMS 系统不提供对应 API，MVP 需要明确降级处理：

| 缺失数据 | PRD 用途 | MVP 降级策略 | 输出标记 |
|----------|----------|-------------|----------|
| 仓库排单/波次信息 | P1 截单时间判断、仓内处理时间估算 | 不检查截单时间，仓内处理时间用固定默认值（24h） | `cutoff_not_checked=true` |
| 仓库当日已接单量 / 日处理产能 | P1 仓容量检查、P2 容量评分 | 如果 API 返回了就用，否则容量维度得分=1.0 | `capacity_estimated=true` |
| 天气状况 | ETA 风险修正（f_weather） | 不叠加天气风险因子，ETA 使用基础估算值 | `weather_not_factored=true` |
| 物流拥堵指数 | ETA 风险修正（f_congestion） | 不叠加拥堵因子 | `congestion_not_factored=true` |
| 承运商准点率 | P2 稳定性评分、ETA 风险修正 | MVP 不含稳定性维度，不修正 ETA | — |
| 承运商揽收班次 | ETA 中的 T_handover 计算 | 交接时间用固定默认值（4h） | `handover_estimated=true` |
| 承运商余额 | 承运商可用性检查 | 不检查余额，假设充足 | `balance_not_checked=true` |
| 承运商实时运费 API | P2 成本评分 | 用距离线性估算替代 | `cost_estimated=true` |
| 承运商实时 ETA API | P2 时效评分 | 用距离线性估算替代 | `eta_estimated=true` |
| SKU 温区属性 | P0 温区匹配 | 无温区标记默认常温，跳过检查 | `temp_zone_defaulted=true` |
| SKU 危险品/超大件标记 | P0 合规/物理可发检查 | 跳过这两项检查 | — |
| 调拨通道信息 | 调拨合发方案 | MVP 不做调拨 | — |

降级标记汇总规则：
- 0 个降级标记 → confidence=high
- 1~3 个降级标记 → confidence=medium
- 4 个以上降级标记 → confidence=low

白盒解释中必须提及关键降级项，例如：
> "推荐从 Valley View 发货（置信度 medium）。注意：库存使用实物库存近似，未扣除预占和锁定；时效为距离估算，未考虑天气和物流拥堵因素。"

## H. 行业前沿能力差距分析（2025-2026 调研）

以下能力在 PRD 中未覆盖，基于 Manhattan Active OMS、Shopify SOR、ShipBob DOM 和 Agentic AI sourcing 的最新实践补充。

### H.1 库存老化/周转率优先发货

传统分仓只看"哪个仓最近"或"哪个仓成本最低"，但先进系统会看"哪个仓的这个 SKU 卖得最慢"。优先从滞销仓发货，减少该仓库存最终被打折清仓的概率。

实现方式：在 P2 评分中加入 `S_velocity` 维度，`S_velocity(w) = 1 - normalize(sell_through_rate(w, sku))`，卖得越慢的仓得分越高（越应该优先发）。

数据依赖：每个 SKU 在每个仓的近 30 天销售速度（日均出库量）。

### H.2 在途库存/预期到货

不只看当前 on-hand，还看"明天有一批补货到仓 A"。如果仓 A 当前缺货但明天到货，可以把订单分到仓 A 等补货到了再发，而不是从远处的仓 B 发。

实现方式：在 P0 库存检查中，`effective_qty = on_hand_qty + inbound_qty_within_N_hours`。

数据依赖：在途采购单/调拨单的预计到货时间和数量。

### H.3 退货回流预判

分析 RMA 数据，预判即将退回的库存。如果某仓有 10 件 RMA 在途，可以纳入可用库存计算。

数据依赖：RMA 状态和预计退回时间。

### H.4 批量订单全局最优（蓄水池模式）

单订单寻仓是局部最优，可能导致所有订单都涌向同一个"最优仓"。蓄水池模式是把一个时间窗口内的订单攒起来一起求解，找全局最优分配。

PRD 已定义 `PoolingWindowSeconds=30`，但 MVP 不实现。

### H.5 Markdown 规避寻仓（Agentic AI 2026 趋势）

这是 2026 年最前沿的概念。AI Agent 在分仓时不只看物流维度，还看商品在各仓的 sell-through velocity 和 markdown 风险。如果某仓的某 SKU 即将进入打折周期，优先从该仓发货消化库存。

参考：[RTInsights - Optimizing Order Sourcing Through the Agentic Shift](https://www.rtinsights.com/optimizing-order-sourcing-for-markdown-avoidance-through-the-agentic-shift/)

### H.6 动态权重自适应

固定权重（w_cost=0.40, w_eta=0.35）在所有场景下都一样，但实际上：
- 生鲜品类应该时效权重更高
- 大促期间应该容量权重更高
- VIP 客户应该时效权重更高
- 低价值订单应该成本权重更高

先进系统会根据订单特征自动调整权重，而非依赖人工配置。

### H.7 历史分仓成功率

某仓对某 SKU 的历史分仓成功率（过去 30 天分了多少单、有多少因库存不足被退回）。成功率低的仓在评分中应该被惩罚。

这比 PRD 的 P3 稳定性规则更细粒度 — P3 看的是仓级别的整体稳定性，这个看的是 SKU×仓 级别的成功率。
