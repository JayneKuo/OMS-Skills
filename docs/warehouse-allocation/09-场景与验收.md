# 场景与验收

本手册覆盖分仓决策引擎所有可能遇到的场景，每个场景均包含：是否正常、最终决策、系统动作、建议怎么办、案例。所有场景必须有标准化最终决策，禁止出现未分类的 catch-all 异常。

---

## 10.1 正常场景

### N1：单仓直发（最常见）

- **是否正常**：正常
- **最终决策**：自动分配至最优单仓，直接下发 WMS
- **系统动作**：P0→P3 漏斗通过 → 单仓方案成本最优 → 装箱校验通过 → ETA 满足 SLA → 财务检查通过 → 库存预占成功 → 下发 WMS
- **建议怎么办**：无需人工干预，系统全自动完成
- **案例**：客户下单 2 件常温商品，华东仓有货、成本最低、ETA 满足次日达承诺，系统自动分配华东仓，顺丰承运，预计次日 18:00 前送达

### N2：多仓拆发

- **是否正常**：正常
- **最终决策**：自动拆分为多个子包裹，分别从不同仓发出
- **系统动作**：单仓无法满足全部 SKU → 枚举拆单方案 → 计算拆单惩罚后综合成本仍优于调拨 → 各子包裹分别通过装箱/ETA/财务检查 → 分别预占库存 → 分别下发 WMS
- **建议怎么办**：无需人工干预；前端展示拆单原因和各包裹独立 ETA
- **案例**：客户下单 A（华东仓有货）+ B（华南仓有货），单仓均无法满足全部，拆为两个包裹分别发出，总成本含拆单惩罚后仍低于调拨合发方案

### N3：调拨合发

- **是否正常**：正常
- **最终决策**：先从源仓调拨至目标仓，合并后统一发出
- **系统动作**：拆单成本过高或客户要求合单 → 计算调拨时效和成本 → 调拨 ETA + 发货 ETA 仍满足 SLA → 触发调拨指令 → 目标仓收货后合并发出
- **建议怎么办**：无需人工干预；前端展示调拨进度和预计合并发货时间
- **案例**：客户下单 3 件商品，仅华东仓有 A+B，华南仓有 C，调拨 C 至华东仓合并发出，总成本低于拆发，且 ETA 仍满足 3 日达承诺

## 10.2 自动降级场景

### D1：SLA 放宽降级

- **是否正常**：降级（非异常，但非最优）
- **最终决策**：放宽时效约束后重新求解，输出次优方案并标记为"降级"
- **系统动作**：P1 阶段所有候选仓的承运商时效均不满足原始 SLA → 自动将 SLA 放宽一档（如次日达降级为隔日达）→ 重新执行 P1→P3 → 输出方案并标记 `degraded = true`
- **建议怎么办**：系统自动完成；前端展示降级原因（"原时效无可用仓，已自动放宽至隔日达"）；通知客户时效变更
- **案例**：客户选择次日达，但所有满足库存的仓的承运商 ETA 均超过次日达阈值，系统自动降级为隔日达，选择成本最优仓发出

### D2：求解超时降级

- **是否正常**：降级（非异常，但非最优）
- **最终决策**：求解超时后使用兜底规则（最近仓 + 默认承运商）输出方案
- **系统动作**：方案枚举/求解在 30 秒内未完成 → 触发超时降级 → 使用兜底规则：选择距离收货地址最近且有货的仓 + 该仓默认承运商 → 输出方案并标记 `degraded = true, reason = SOLVER_TIMEOUT`
- **建议怎么办**：系统自动完成；技术团队需关注超时频率，若超时率 > 5% 需优化求解性能
- **案例**：大促期间订单量激增，求解器在 30 秒内未完成全量方案枚举，系统自动降级为最近仓直发方案

## 10.3 挂起类场景

### H1：部分 SKU 缺货（部分发货）

- **是否正常**：挂起
- **最终决策**：可发 SKU 先行发出，缺货 SKU 挂起等待补货或取消
- **系统动作**：订单含多个 SKU → 部分 SKU 全网无货 → 拆分为"可发子单"和"缺货子单" → 可发子单正常走决策流程 → 缺货子单进入挂起池，设置补货监听
- **建议怎么办**：通知客户部分商品缺货，确认是否接受部分发货；若客户拒绝，整单挂起
- **案例**：客户下单 A+B+C，其中 C 全网无货，系统将 A+B 拆为可发子单正常发出，C 挂起等待补货

#### H1-1 部分发货边界规则

部分发货需满足以下边界条件，否则整单挂起：

| 规则 | 说明 |
|------|------|
| 可发金额占比 ≥ 50% | 可发 SKU 的金额合计必须占订单总金额的 50% 以上，否则不拆单，整单挂起 |
| 可发件数 ≥ 1 | 至少有 1 个 SKU 可发，否则整单挂起 |
| 客户未明确拒绝拆单 | 若客户在下单时选择"不接受部分发货"，则整单挂起 |
| 缺货 SKU 非主商品 | 若缺货 SKU 为订单主商品（金额最高的 SKU），整单挂起 |

#### 部分发货 SLA 规则

| 规则 | 说明 |
|------|------|
| 可发子单 SLA 不变 | 可发部分按原始 SLA 执行，不因拆单而放宽 |
| 缺货子单 SLA 重置 | 缺货部分的 SLA 从补货到仓时间重新计算 |
| 合并发货窗口 | 若缺货 SKU 预计 24 小时内到货，系统等待合并发货而非拆单 |

#### 部分发货超时处理

| 规则 | 说明 |
|------|------|
| 缺货挂起超过 48 小时 | 自动通知客户，建议取消缺货部分或更换商品 |
| 缺货挂起超过 72 小时 | 自动取消缺货子单，退款处理 |
| 补货到仓触发 | 缺货 SKU 补货到仓后，自动唤醒挂起子单，重新进入决策流程 |

### H2：地址解析模糊

- **是否正常**：挂起
- **最终决策**：订单挂起至异常池，等待人工补全或客户确认地址
- **系统动作**：地址解析服务返回置信度 < 阈值（默认 0.8）→ 无法确定配送区域 → 订单挂起，状态置为 ADDR_AMBIGUOUS → 推送至异常池
- **建议怎么办**：客服联系客户确认详细地址；地址补全后自动重新进入决策流程
- **案例**：客户填写地址为"XX 市 XX 路"，缺少门牌号和区，解析置信度 0.6，系统挂起订单并通知客服补全

### H3：承运商全部不可用

- **是否正常**：挂起
- **最终决策**：订单挂起，等待承运商恢复或人工指定替代承运商
- **系统动作**：所有候选承运商均不可用（停运/余额不足/超负荷）→ 订单挂起，状态置为 CARRIER_UNAVAILABLE → 设置承运商状态监听，恢复后自动重试
- **建议怎么办**：运营检查承运商状态；若短期无法恢复，人工指定替代承运商或切换仓库
- **案例**：台风天气导致华南区域所有承运商停运，相关订单全部挂起，天气恢复后自动重试

### H4：仓容量全部超限

- **是否正常**：挂起
- **最终决策**：订单进入蓄水池排队，等待仓容量释放后按优先级重新分配
- **系统动作**：所有候选仓当日容量均已超限 → 订单进入蓄水池 → 按订单优先级（加急 > 普通）排队 → 仓容量释放后自动触发重算
- **建议怎么办**：大促前提前扩容；蓄水池订单超过阈值时触发告警，运营评估是否需要临时扩容
- **案例**：双十一期间所有仓容量打满，新订单进入蓄水池排队，每 5 分钟检查一次容量释放情况

### H5：装箱失败

- **是否正常**：挂起
- **最终决策**：订单挂起，等待人工确认装箱方案或调整商品组合
- **系统动作**：装箱计算失败（超长超重/禁混冲突/无合适箱型）→ 尝试拆包重算 → 仍失败 → 订单挂起，状态置为 CARTON_FAILED → 推送至异常池
- **建议怎么办**：仓库主管确认是否可以特殊包装；若商品主数据有误（尺寸/重量），通知商家修正
- **案例**：客户下单一件超长商品（2.5m）和一件易碎品，禁混规则不允许同箱，且超长商品无匹配箱型，系统挂起等待人工处理

### H6：库存预占反复失败

- **是否正常**：挂起
- **最终决策**：订单挂起至异常池，等待库存恢复或人工介入
- **系统动作**：库存预占重试 3 次均失败（并发竞争/库存数据不一致）→ 排除当前仓重算 → 所有仓均预占失败 → 订单挂起，状态置为 LOCK_EXHAUSTED → 推送至异常池
- **建议怎么办**：技术团队检查库存一致性；运营确认是否存在超卖；若库存数据异常，触发库存盘点
- **案例**：大促秒杀场景，某爆款 SKU 并发预占导致反复失败，系统在 3 次重试 + 换仓重算后仍无法预占，订单挂起

## 10.4 阻断类场景

### B1：地址解析失败

- **是否正常**：阻断
- **最终决策**：订单阻断，无法进入决策流程
- **系统动作**：地址解析服务返回失败（无法识别/格式错误/海外地址不支持）→ 订单阻断，错误码 ADDR_PARSE_FAILED → 推送至异常池
- **建议怎么办**：客服联系客户重新填写地址；若为系统性解析失败，技术团队排查地址服务
- **案例**：客户填写地址为纯英文且非系统支持的海外地址格式，解析服务返回失败，订单阻断

### B2：SKU 主数据缺失

- **是否正常**：阻断
- **最终决策**：订单阻断，等待商家补全商品主数据
- **系统动作**：SKU 缺少必要属性（重量/尺寸/温区/品类）→ 无法进行装箱计算和仓匹配 → 订单阻断，错误码 SKU_DATA_MISSING → 通知商家
- **建议怎么办**：商家在商品管理后台补全缺失字段；补全后订单自动重新进入决策流程
- **案例**：新上架商品未填写重量和尺寸，系统无法计算运费和装箱方案，订单阻断

### B3：风控拦截

- **是否正常**：阻断
- **最终决策**：订单阻断，转风控团队处理
- **系统动作**：命中黑名单/恶意刷单/欺诈检测规则 → 订单阻断，错误码 RISK_BLOCKED → 生成风控工单
- **建议怎么办**：风控团队审核；若为误判，解除拦截后订单自动重新进入决策流程
- **案例**：同一收货地址 1 小时内下单 50 次，命中刷单规则，全部订单阻断转风控审核

### B4：全网无候选仓

- **是否正常**：阻断
- **最终决策**：订单阻断，无可用仓库
- **系统动作**：P0 硬约束过滤后候选集为空 → 所有仓均被淘汰 → 订单阻断，错误码 NO_CANDIDATE_WAREHOUSE → 记录各仓淘汰原因
- **建议怎么办**：运营根据各仓淘汰原因排查（库存不足/配送范围不覆盖/温区不匹配等）；若为库存问题，触发补货流程
- **案例**：客户下单冷冻商品，但收货地址所在区域无冷链仓覆盖，所有仓在 P0-6（温区匹配）被淘汰

### B5：财务熔断被拒

- **是否正常**：阻断
- **最终决策**：订单阻断，财务审批被拒绝
- **系统动作**：触发财务熔断 → 进入审批队列 → 审批人拒绝 → 订单阻断，错误码 FINANCE_REJECTED → 释放预占资源
- **建议怎么办**：商家调整商品定价或运费策略；运营评估是否需要调整熔断阈值
- **案例**：某订单运费占客单价 80%（阈值 50%），触发熔断，审批人判断为不合理订单，拒绝审批

### B6：承运商余额不足

- **是否正常**：阻断
- **最终决策**：该承运商所有待发订单阻断，等待充值
- **系统动作**：承运商账户余额 < 最低阈值 → 停止该承运商所有后续自动下发 → 已分配该承运商的待发订单阻断，错误码 CARRIER_BALANCE_LOW → 触发告警
- **建议怎么办**：财务团队立即充值；运营将待发订单切换至其他承运商
- **案例**：顺丰账户余额不足 1000 元，系统自动停止所有顺丰待发订单，通知财务充值

### B7：合规限制

- **是否正常**：阻断
- **最终决策**：订单阻断，商品在目标区域禁售
- **系统动作**：商品命中目标区域的禁售/限售清单（跨境限制/特殊品类管控/区域政策）→ 订单阻断，错误码 COMPLIANCE_BLOCKED → 通知商家
- **建议怎么办**：商家确认商品合规性；若为区域限制，建议客户更换收货地址或取消订单
- **案例**：客户从海外仓购买含锂电池商品，目标国海关限制进口，订单阻断

### B8：WMS 拦截失败

- **是否正常**：阻断
- **最终决策**：订单阻断，转人工处理
- **系统动作**：逆向事件触发 WMS 拦截 → WMS 返回拦截失败（已拣货/已打包/已交接）→ 订单阻断，错误码 CANCEL_INTERCEPT_FAILED → 冻结该订单后续自动操作 → 推送至人工处理队列
- **建议怎么办**：仓库主管确认实物状态；若已出库，走逆向物流退回；若未出库，手动拦截
- **案例**：客户下单后 30 分钟取消，但 WMS 已完成拣货打包，拦截失败，转人工处理决定是否追回包裹

## 10.5 自动重算与逆向场景

### R1：库存预占失败触发重算

- **是否正常**：自动重算
- **最终决策**：排除失败仓，自动重新求解
- **系统动作**：库存预占失败 → 将失败仓加入本次排除名单 → 重新执行 P0→P3 + 方案求解 → 最多重试 3 次 → 仍失败则挂起
- **建议怎么办**：无需人工干预（重试次数内）；若反复失败，技术团队检查库存一致性
- **案例**：华东仓库存预占失败（并发竞争），系统自动排除华东仓，重新求解后分配至华北仓

### R2：WMS 下发失败触发重算

- **是否正常**：自动重算
- **最终决策**：释放预占库存，排除失败仓，自动重新求解
- **系统动作**：WMS 下发失败（WMS 系统异常/接口超时）→ 释放已预占库存 → 将失败仓加入排除名单 → 重新执行完整决策流程
- **建议怎么办**：无需人工干预；若同一仓 WMS 反复失败，触发仓健康度告警
- **案例**：华南仓 WMS 系统升级中，下发失败，系统自动释放库存并切换至华东仓重新求解

### R3：用户修改地址触发重算

- **是否正常**：逆向重算
- **最终决策**：根据当前状态决定处置方式，成功拦截后重新求解
- **系统动作**：用户修改收货地址 → 检查当前订单状态 → 若在运算中/待审核：直接作废重算 → 若在已锁库/已下发：发起 WMS 拦截 → 拦截成功后释放资源，以新地址重新进入决策流程
- **建议怎么办**：系统自动处理；若 WMS 拦截失败，转人工处理
- **案例**：客户下单后 10 分钟修改地址从北京改为上海，订单处于已锁库状态，系统发起 WMS 拦截成功，释放华北仓库存，以新地址重新求解后分配至华东仓

### R4：用户取消订单

- **是否正常**：逆向取消
- **最终决策**：根据当前状态决定处置方式，尽可能自动完成取消
- **系统动作**：用户取消订单 → 检查当前订单状态 → 若在运算中/待审核：直接作废，释放所有资源，状态置为 CANCELLED → 若在已锁库/已下发：发起 WMS 拦截 → 拦截成功：释放库存，状态置为 CANCELLED → 拦截失败：转人工处理
- **建议怎么办**：系统自动处理；拦截失败时客服告知客户可能需要拒收后走退货流程
- **案例**：客户下单后 5 分钟取消，订单处于待审核状态，系统直接作废决策，释放所有预占资源，订单取消成功
