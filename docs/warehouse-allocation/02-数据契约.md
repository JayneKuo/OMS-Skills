# 数据契约

## 5.1 订单输入最小闭集

## 5.5 数据缺失策略

当输入数据缺失时，系统按以下策略处理，不允许静默忽略。

| 缺失数据 | 策略 | 说明 |
|----------|------|------|
| order_id 缺失 | 阻断 | 直接拒绝请求，返回参数错误 |
| request_idempotency_key 缺失 | 阻断 | 直接拒绝请求，无法保证幂等性 |
| shipping_address 缺失 | 阻断 | 无法计算 ETA 和运费，阻断并创建地址异常工单 |
| shipping_address 无法解析 | 阻断 | 地址解析失败，阻断并提示人工确认地址 |
| SKU 重量/尺寸缺失 | 降级 | 使用品类平均值替代，标记为"数据降级"，决策置信度降低 |
| SKU temperature_zone 缺失 | 默认常温 | 按常温处理，标记为"数据降级" |
| SKU hazmat_type 缺失 | 默认无 | 按非危险品处理，标记为"数据降级" |
| 仓库库存数据过期（>10分钟） | 降级 | 使用过期数据但标记为"库存数据陈旧"，决策置信度降低 |
| 仓库库存数据缺失 | 排除该仓 | 该仓不进入候选集，记录排除原因 |
| 承运商 ETA API 不可用 | 降级 | 使用历史 ETA 数据替代，标记为"ETA 降级" |
| 承运商历史准点率缺失 | 默认 0.8 | 使用默认准点率 80%，标记为"数据降级" |
| 承运商余额数据缺失 | 降级 | 假设余额充足，但标记为"余额未校验"，决策置信度降低 |
| 承运商价格数据缺失 | 排除该承运商 | 该承运商不进入候选集，记录排除原因 |
| customer_level 缺失 | 默认普通 | 按普通客户处理 |
| promised_sla 缺失 | 默认标准时效 | 按标准时效处理 |
| allow_split 缺失 | 默认 true | 允许拆单 |
| max_package_count 缺失 | 默认 5 | 最多拆 5 个包裹 |

## 6.2 客户体验策略定义

以下 5 个策略字段控制分仓决策中的客户体验偏好，由订单级传入或从客户等级规则继承。

### 策略 1：customer_experience_penalty（客户体验惩罚系数）

- 取值范围：0.0 ~ 1.0
- 默认值：0.0（普通客户），0.3（VIP），0.6（SVIP）
- 作用：值越高，系统越倾向于选择 ETA 更短的方案，即使成本更高
- 影响：直接叠加到最终评分公式中，增大时效维度的权重
- 示例：SVIP 客户下单，系统会优先选择次日达方案，即使运费高出 30%

### 策略 2：gift_same_package_required（赠品必须同包）

- 取值范围：true / false
- 默认值：true
- 作用：为 true 时，赠品 SKU 必须与主商品放在同一个包裹中
- 影响：限制装箱方案的自由度，可能导致无法拆单或必须选择特定仓
- 示例：买洗发水送小样，小样必须和洗发水放在同一个包裹里

### 策略 3：bundle_same_warehouse_required（组合商品必须同仓）

- 取值范围：true / false
- 默认值：true
- 作用：为 true 时，组合商品的所有子 SKU 必须从同一个仓库发出
- 影响：缩小候选仓范围，只有同时有所有子 SKU 库存的仓才能入选
- 示例：套装礼盒包含 3 个 SKU，必须从同一个仓发出，不可拆开

### 策略 4：partial_fulfillment_allowed（允许部分履约）

- 取值范围：true / false
- 默认值：false
- 作用：为 true 时，库存不足的 SKU 可以先发有货部分，缺货部分挂起等补货
- 影响：增加方案灵活性，但可能导致客户收到多次发货
- 示例：订单含 3 个 SKU，其中 1 个缺货，先发 2 个，缺货的等补货后再发

### 策略 5：high_value_force_single_delivery（高价值订单强制单包）

- 取值范围：true / false
- 默认值：false（订单金额超过 HighValueOrderThreshold 时自动设为 true）
- 作用：为 true 时，订单强制从单一仓库、单一包裹发出
- 影响：禁止拆单，缩小候选仓范围，可能增加成本但降低丢件风险
- 示例：订单金额 8000 元，强制单仓单包发出，降低理赔风险

## 6.4 算法参数中文解释

| 参数 | 中文含义 | 作用说明 |
|------|----------|----------|
| Alpha (α) | 成本权重 | 控制综合评分中"成本"维度的重要程度。值越大，系统越倾向于选择低成本方案 |
| Beta (β) | 时效权重 | 控制综合评分中"时效"维度的重要程度。值越大，系统越倾向于选择快速送达方案 |
| Gamma (γ) | 风险权重 | 控制综合评分中"风险"维度的重要程度。值越大，系统越倾向于规避高风险方案 |
| Delta (δ) | 容量权重 | 控制综合评分中"仓容量"维度的重要程度。值越大，系统越倾向于均衡仓网负荷 |
| SplitPenaltyFactor | 拆单惩罚系数 | 每多拆一个包裹的额外成本惩罚（元）。值越大，系统越不愿意拆单 |
| CapacityPenaltyFactor | 容量惩罚系数 | 仓负荷超阈值后的惩罚力度。值越大，系统越积极地将订单分散到其他仓 |
| ETARiskBuffer | ETA 风险缓冲 | ETA 校准时的安全余量。值越大，预估送达时间越保守 |
| OntimeRateThreshold | 准点率阈值 | 承运商准点率的最低可接受值。低于此值时不信任其 ETA API |
| MaxSolverTimeout | 求解器超时 | 方案枚举的最大计算时间。超时后使用已有最优解或降级策略 |
| GrossMarginRedLine | 毛利率红线 | 触发财务熔断的毛利率底线。低于此值阻断自动下发 |
| ConfidenceThreshold | 置信度阈值 | 决策置信度的最低可接受值。低于此值建议人工复核 |

## 7.10.1 可用库存公式

**人话解释**：可用库存 = 仓库里实际有的货 − 已经被别的订单占住的 − 已经锁定准备发货的 − 不能动的安全库存。

```
available_qty = physical_qty - reserved_qty - locked_qty - safety_stock
```

| 符号 | 中文名 | 说明 |
|------|--------|------|
| available_qty | 可用库存数量 | 当前可以分配给新订单的库存数量 |
| physical_qty | 实物库存数量 | 仓库中该 SKU 的实际在库数量（含所有状态） |
| reserved_qty | 预占数量 | 已被分仓引擎预占（Phase 1）但尚未正式锁定的数量 |
| locked_qty | 锁定数量 | 已正式锁定（Phase 2）等待 WMS 拣货的数量 |
| safety_stock | 安全库存 | 为防止缺货而预留的最低库存量，不参与正常分配 |

**业务含义**：只有 available_qty > 0 的仓才能作为候选仓。如果 available_qty ≤ 0，说明该仓的库存已经全部被占用或低于安全线，不能再接新订单。

## 7.10.2 ETA 总公式

**人话解释**：预估送达时间 = 排队等多久 + 等截单多久 + 仓库处理多久 + 等快递来取多久 + 路上运多久 + 最后一公里送多久 + 天气耽误多久 + 预留的风险缓冲。

```
ETA_total = T_queue + T_cutoff_wait + T_process + T_handover + T_transit + T_last_mile + T_weather + T_risk_buffer
```

| 符号 | 中文名 | 说明 |
|------|--------|------|
| ETA_total | 预估总送达时间 | 从下单到客户签收的预估总时间（小时） |
| T_queue | 排队等待时间 | 订单在仓库作业队列中等待的时间，取决于仓当前积压量 |
| T_cutoff_wait | 截单等待时间 | 如果下单时间晚于截单时间，需要等到下一个作业窗口 |
| T_process | 仓内处理时间 | 拣货 + 复核 + 打包 + 称重 + 贴面单的总时间 |
| T_handover | 交接时间 | 仓库打包完成到承运商实际揽收之间的等待时间 |
| T_transit | 干线运输时间 | 承运商从揽收到目的地转运中心的运输时间 |
| T_last_mile | 末端配送时间 | 从目的地转运中心到客户签收的配送时间 |
| T_weather | 天气影响时间 | 极端天气（暴雨、暴雪、台风）导致的额外延迟 |
| T_risk_buffer | 风险缓冲时间 | 基于历史数据计算的风险缓冲，覆盖不可预见的延迟 |

**业务含义**：ETA 不是简单地问快递公司"几天能到"，而是把从下单到签收的每一段时间都拆开算，这样才能准确预估。特别是截单时间和仓库积压这两个因素，很多系统会忽略，导致 ETA 过于乐观。

---

## 7.10.3 ETA 风险修正公式

**人话解释**：遇到坏天气、物流拥堵、快递公司不靠谱这些情况，要在基础 ETA 上加一个风险系数，让预估时间更保守、更靠谱。

```
ETA_adjusted = ETA_base × (1 + risk_factor)
```

```
risk_factor = max(f_weather, f_congestion, f_carrier_risk)
```

| 符号 | 中文名 | 说明 |
|------|--------|------|
| ETA_adjusted | 风险修正后 ETA | 叠加风险因子后的预估送达时间 |
| ETA_base | 基础 ETA | 未叠加风险因子的原始预估送达时间 |
| risk_factor | 风险因子 | 取天气、拥堵、承运商风险中的最大值 |
| f_weather | 天气风险因子 | 暴雨 0.3，暴雪/台风 0.5 |
| f_congestion | 拥堵风险因子 | 普通大促 0.2，双 11 高峰 0.5 |
| f_carrier_risk | 承运商风险因子 | (1 − 准点率) × 0.5，准点率越低风险越大 |

**业务含义**：风险因子取最大值而不是累加，是因为这些风险通常不会同时叠加到最大程度。比如暴雪天气本身就会导致物流拥堵，两者有重叠。

---

## 7.10.4 综合成本公式

**人话解释**：总成本 = 快递费 + 仓库操作费 + 调拨费（如果有）+ 拆单罚款 + 仓库太挤的罚款 + 风险预估成本。把所有花的钱和可能花的钱加在一起，才能公平比较不同方案。

```
Cost_total = Freight_order + Cost_warehouse + Cost_transfer + Penalty_split + Penalty_capacity + Cost_risk
```

| 符号 | 中文名 | 说明 |
|------|--------|------|
| Cost_total | 综合总成本 | 该方案的所有成本之和（人民币元） |
| Freight_order | 订单运费 | 所有包裹的运费之和 |
| Cost_warehouse | 仓操费 | 仓库拣货、打包等操作费用 |
| Cost_transfer | 调拨费 | 调拨合发场景下的调拨运输和操作费用，非调拨方案为 0 |
| Penalty_split | 拆单惩罚 | 拆单带来的额外成本，= (拆单仓数 − 1) × 单位惩罚金额 |
| Penalty_capacity | 容量惩罚 | 仓容量接近上限时的惩罚，防止某个仓被打爆 |
| Cost_risk | 风险成本 | 基于时效风险和承运商风险的额外成本预估 |

**业务含义**：拆单惩罚（Penalty_split）是关键设计——如果不加这个惩罚，算法会倾向于把订单拆到多个仓分别发货（因为每个仓可能运费更低），但实际上拆单会增加包装成本、客户体验变差、售后复杂度增加。容量惩罚（Penalty_capacity）则是为了防止所有订单都涌向同一个"最优仓"，导致该仓爆仓。

---

## 7.10.5 综合评分公式

**人话解释**：最终评分 = 成本分 × 成本权重 + 时效分 × 时效权重 + 准时率 × 准时权重 + 容量分 × 容量权重。四个维度加权求和，得到一个总分来排序。

```
Score = w_cost × Normalize(Cost_total) + w_eta × Normalize(ETA) + w_ontime × OnTimeProbability + w_cap × Normalize(RemainCapacity)
```

| 符号 | 中文名 | 默认值 | 说明 |
|------|--------|--------|------|
| Score | 综合评分 | — | 该方案的最终评分，越高越优 |
| w_cost | 成本权重 | 0.40 | 成本在总评分中的占比 |
| w_eta | 时效权重 | 0.30 | 时效在总评分中的占比 |
| w_ontime | 准时率权重 | 0.15 | 准时送达概率在总评分中的占比 |
| w_cap | 容量权重 | 0.15 | 仓剩余容量在总评分中的占比 |
| Normalize() | 归一化函数 | — | 将不同量纲的值映射到 [0, 1] 区间 |
| OnTimeProbability | 准时送达概率 | — | 该方案在 SLA 时间内送达的概率 |
| RemainCapacity | 剩余容量 | — | 该仓当日剩余可处理订单量 |

**业务含义**：权重决定了"省钱"和"快速"哪个更重要。默认配置下成本占 40%、时效占 30%，说明系统偏向省钱但也重视速度。商家可以根据自己的策略调整权重，比如生鲜商家可以把时效权重调到 0.50。

---

## 7.10.6 候选仓局部评分公式

**人话解释**：每个候选仓的得分 = 成本分 × 成本权重 + 时效分 × 时效权重 + 容量分 × 容量权重 + 稳定性分 × 稳定性权重。用来在方案枚举前先给每个仓打分排序。

```
Score_local(w) = w_cost × S_cost(w) + w_eta × S_eta(w) + w_cap × S_cap(w) + w_stab × S_stab(w)
```

| 符号 | 中文名 | 说明 |
|------|--------|------|
| Score_local(w) | 仓 w 的局部评分 | 该仓在候选仓阶段的综合得分 |
| S_cost(w) | 仓 w 的成本分 | 归一化后的成本得分，越低成本对应越高分数 |
| S_eta(w) | 仓 w 的时效分 | 归一化后的时效得分，越快对应越高分数 |
| S_cap(w) | 仓 w 的容量分 | 归一化后的剩余容量得分 |
| S_stab(w) | 仓 w 的稳定性分 | 归一化后的稳定性得分，基于准点率和异常率 |

**业务含义**：局部评分用于候选仓裁剪阶段，在方案枚举之前先淘汰明显不优的仓，减少后续计算量。它和综合评分（Score）的区别是：局部评分只看单个仓的表现，综合评分看的是整个方案（可能涉及多个仓）的表现。

## 7.10.7 归一化公式

**人话解释**：归一化就是把不同单位的数字（比如"元"和"小时"）统一换算到 0 到 1 之间，这样才能放在一起比较和加权。

成本归一化（越低越好，所以用 1 减）：
```
S_cost(w) = 1 − (Cost(w) − Cost_min) / (Cost_max − Cost_min)
```

时效归一化（越快越好，所以用 1 减）：
```
S_eta(w) = 1 − (ETA(w) − ETA_min) / (ETA_max − ETA_min)
```

容量归一化（越多越好，直接正向）：
```
S_cap(w) = (RemainCap(w) − RemainCap_min) / (RemainCap_max − RemainCap_min)
```

稳定性归一化（越高越好，直接正向）：
```
S_stab(w) = (OnTimeRate(w) − OnTimeRate_min) / (OnTimeRate_max − OnTimeRate_min)
```

特殊情况处理：
```
当 max = min 时，S = 1.0（所有候选仓该维度相同，不产生区分度）
```

| 符号 | 中文名 | 说明 |
|------|--------|------|
| Cost(w) | 仓 w 的成本 | 该仓的综合成本（元） |
| Cost_min / Cost_max | 成本最小值/最大值 | 所有候选仓中成本的最小值和最大值 |
| ETA(w) | 仓 w 的 ETA | 该仓的预估送达时间（小时） |
| ETA_min / ETA_max | ETA 最小值/最大值 | 所有候选仓中 ETA 的最小值和最大值 |
| RemainCap(w) | 仓 w 的剩余容量 | 该仓当日剩余可处理订单量 |
| RemainCap_min / RemainCap_max | 剩余容量最小值/最大值 | 所有候选仓中剩余容量的最小值和最大值 |
| OnTimeRate(w) | 仓 w 的准点率 | 该仓近期的准时发货率 |
| OnTimeRate_min / OnTimeRate_max | 准点率最小值/最大值 | 所有候选仓中准点率的最小值和最大值 |

**业务含义**：归一化的关键是"在候选仓之间相对比较"。比如候选仓 A 成本 20 元、仓 B 成本 30 元，归一化后 A 的成本分 = 1.0（最优），B 的成本分 = 0.0（最差）。如果只有一个候选仓，所有维度归一化后都是 1.0，不产生区分度。

## 7.10.8 运费公式

**人话解释**：运费 = 首重费 + 超出首重的部分按续重单价算。超出部分要向上取整到续重步长的整数倍。

基础运费（首重 + 续重模式）：
```
Freight_base = FirstWeight_fee + ceil((ChargeWeight - FirstWeight) / StepWeight) × StepWeight_fee
```

计费重量：
```
ChargeWeight = max(实际重量, 体积重量)
```

体积重量：
```
体积重量 = (长 × 宽 × 高) / 体积重量系数
```

计费重量取整：
```
ChargeWeight_rounded = ceil(ChargeWeight × 10) / 10
```

订单总运费：
```
Freight_order = Σ(Freight_base(i) + Freight_surcharge(i))  for i in packages
```

| 符号 | 中文名 | 说明 |
|------|--------|------|
| Freight_base | 基础运费 | 按首重续重规则计算的基础运费（元） |
| FirstWeight_fee | 首重费用 | 首重部分的固定费用（元） |
| FirstWeight | 首重重量 | 首重的重量上限（通常 1kg） |
| ChargeWeight | 计费重量 | 取实际重量和体积重量中的较大值（kg） |
| StepWeight | 续重步长 | 续重的计费步长（通常 1kg 或 0.5kg） |
| StepWeight_fee | 续重单价 | 每个续重步长的费用（元） |
| Freight_surcharge | 附加费 | 各类附加费之和（偏远、超重、冷链等） |
| Freight_order | 订单总运费 | 所有包裹运费之和（元） |
| ceil() | 向上取整函数 | 将数值向上取整到最近的整数 |

**业务含义**：运费计算中最容易出错的是"续重取整"。比如计费重量 2.3kg，首重 1kg，续重步长 0.5kg，那么续重段数 = ceil((2.3-1)/0.5) = ceil(2.6) = 3 段，而不是 2.6 段。这个向上取整会直接影响运费金额。

## 7.10.9 容量惩罚公式

**人话解释**：仓库越挤，惩罚越重。70% 以下不罚，70%-85% 轻罚，85%-95% 中罚，95% 以上重罚。这样可以防止所有订单都涌向同一个仓。

```
Penalty_capacity = 
  0                                          当 utilization ∈ [0%, 70%)
  (utilization - 0.70) × 10                  当 utilization ∈ [70%, 85%)
  1.5 + (utilization - 0.85) × 30            当 utilization ∈ [85%, 95%)
  4.5 + (utilization - 0.95) × 100           当 utilization ∈ [95%, 100%]
```

| 符号 | 中文名 | 说明 |
|------|--------|------|
| Penalty_capacity | 容量惩罚 | 因仓容量紧张而增加的惩罚成本（元） |
| utilization | 容量利用率 | 该仓当日已用容量 / 总容量，范围 [0, 1] |

各梯度的惩罚金额示例：

| 利用率 | 惩罚金额 | 说明 |
|--------|----------|------|
| 60% | 0 元 | 容量充足，无惩罚 |
| 75% | 0.5 元 | 轻度拥挤 |
| 80% | 1.0 元 | 轻度拥挤 |
| 90% | 3.0 元 | 中度拥挤 |
| 95% | 4.5 元 | 严重拥挤 |
| 98% | 7.5 元 | 接近满仓 |
| 100% | 9.5 元 | 满仓 |

**业务含义**：容量惩罚是一个"软约束"——它不会直接禁止分配到高利用率的仓，而是通过增加成本来降低该仓的综合评分。这样在其他仓都不太好的情况下，高利用率的仓仍然可以被选中，但需要付出更高的"代价"。惩罚金额的阶梯设计是为了在 85% 以上快速增加惩罚力度，因为超过 85% 后仓库的作业效率会明显下降。

## 7.10.10 调拨成本与 ETA 公式

**人话解释**：调拨就是把缺的货从别的仓运到主仓，再统一发给客户。调拨成本 = 调拨运费 + 两头的操作费 + 包装费 + 货物在路上占用资金的成本。调拨时间 = 源仓处理 + 路上运输 + 主仓收货 + 合单 + 主仓发货。

调拨成本公式：
```
Cost_transfer = Freight_transfer + Cost_source_out + Cost_main_in + Cost_packing + Cost_inventory_hold
```

调拨 ETA 公式：
```
ETA_transfer = T_source_process + T_transfer_transit + T_main_receive + T_main_merge + T_main_ship
```

| 符号 | 中文名 | 说明 |
|------|--------|------|
| Cost_transfer | 调拨总成本 | 调拨方案的所有额外成本之和（元） |
| Freight_transfer | 调拨运费 | 源仓到主仓的运输费用 |
| Cost_source_out | 源仓出库费 | 源仓拣货、打包、出库的操作费用 |
| Cost_main_in | 主仓入库费 | 主仓接收、验收、上架的操作费用 |
| Cost_packing | 包装材料费 | 调拨包装的材料费用 |
| Cost_inventory_hold | 库存占用成本 | 调拨在途期间的资金占用成本，= 商品价值 × 日利率 × 在途天数 |
| ETA_transfer | 调拨总时间 | 从源仓开始处理到主仓发货完成的总时间（小时） |
| T_source_process | 源仓处理时间 | 源仓拣货、打包、出库的时间 |
| T_transfer_transit | 调拨运输时间 | 源仓到主仓的运输时间 |
| T_main_receive | 主仓收货时间 | 主仓接收、验收、上架的时间 |
| T_main_merge | 主仓合单时间 | 将调拨商品与本仓商品合并打包的时间 |
| T_main_ship | 主仓发货时间 | 合并后的正常发货流程时间 |

**业务含义**：调拨合发的核心权衡是"时间换体验"——客户只收到一个包裹（体验好），但需要等调拨完成（时间长）。系统会自动比较调拨合发和多仓拆发的成本与时效，只有在调拨方案明显更优时才会推荐调拨。库存占用成本虽然金额通常很小，但在高价值商品（如电子产品）的长距离调拨中不可忽略。
