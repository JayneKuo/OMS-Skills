# 需求总览

### 1.1 业务定位

智能分仓决策引擎是基于多目标运筹优化的自动化履约大脑。它不是"最近仓规则集合"，也不是"有货仓查询器"，而是一套把订单拆解为库存约束、装箱约束、时效约束、仓网负荷约束、承运能力约束、财务约束、风险约束的统一求解系统，并在这些约束下输出唯一、可执行、可解释、可补偿的最终履约方案。

### 1.2 核心目标

1. 在满足合规、安全、物理可履约、库存可用的前提下，输出最优履约方案。
2. 在满足客户 SLA 的前提下，尽可能降低总履约成本。
3. 控制仓网拥塞，避免局部仓持续被打爆。
4. 保证高并发下库存一致性、执行幂等性与失败补偿闭环。
5. 所有正常、异常、边界场景都必须有标准化最终决策、系统动作、人工建议和案例。

### 1.3 本期不做

1. 自动采购
2. 自动创建 PO
3. 自动修复商品主数据
4. 自动修改商家规则配置
5. 在线机器学习闭环调参

### 1.4 一句话定义

智能分仓决策引擎是一套将库存、装箱、时效、成本、风险、财务与人工边界统一纳入计算，并对白盒输出最终履约决策与兜底动作的履约中台。

### 1.5 阅读约定

为了让这份文档可以"一遍读懂"，本文统一按以下方式写：

1. 英文字段第一次出现时，尽量直接给出中文含义。
2. 英文缩写和公式，在前面先讲"人话解释"，后面再给规则细节。
3. 你在后文看到的很多英文名，本质上都是系统字段名，不是额外知识点。
4. 如果你只想先读懂方案，不需要先懂代码或算法，可以先看本章，再继续往下读正文。

### 1.6 一页读懂核心术语

| 术语 | 中文含义 | 说明 |
|------|----------|------|
| ATP | 可承诺库存（Available To Promise） | 物理库存 − 安全库存 − 已预占 − 已锁定。只有 ATP > 0 的仓才进入候选集 |
| SafetyStock | 安全库存 | 每个 SKU 在每个仓的最低保留量，不参与正常分配 |
| allow_split | 是否允许拆单 | 订单级开关。为 false 时，所有 SKU 必须从同一仓发出 |
| max_package_count | 最大包裹数 | 单笔订单允许拆出的最大包裹数，超过则方案不合法 |
| temperature_zone | 温区 | SKU 属性：常温 / 冷藏 / 冷冻。同一包裹内 SKU 温区必须兼容 |
| hazmat_type | 危险品类型 | SKU 属性：易燃 / 易爆 / 腐蚀 / 无。决定能否混装、能否空运 |
| must_ship_with | 必须同包 | SKU 级绑定关系，指定哪些 SKU 必须放在同一个包裹 |
| cannot_ship_with | 禁止同包 | SKU 级互斥关系，指定哪些 SKU 不能放在同一个包裹 |
| oversize_flag | 超大件标记 | SKU 属性。超大件走专线物流，不进普通包裹 |
| ETA | 预计送达时间（Estimated Time of Arrival） | 从下单到客户签收的端到端时间 |
| ETA_P50 | ETA 中位数 | 历史数据中 50% 订单能在此时间内送达 |
| ETA_P90 | ETA 第 90 百分位 | 历史数据中 90% 订单能在此时间内送达，用于风险化口径 |
| OnTimeProbability | 准时送达概率 | 基于历史数据计算的该仓→该地址在承诺时效内送达的概率 |
| Freight | 运费 | 单个包裹的承运商运费，基于计费重和阶梯价计算 |
| PackagingCost | 包材成本 | 单个包裹的包装材料费用 |
| OpsCost | 操作成本 | 单个包裹的仓内拣货、打包、出库操作费 |
| SplitPenalty | 拆单惩罚 | 拆单时额外附加的成本惩罚，防止过度拆单 |
| TransferCost | 调拨成本 | 从 A 仓调货到 B 仓的运输 + 操作成本 |
| CapacityPenalty | 容量惩罚 | 仓库负荷超过阈值时的额外成本惩罚，用于均衡仓网负荷 |
| InventoryOpportunityCost | 库存机会成本 | 占用某仓库存导致其他订单无法履约的隐性成本 |
| decision_code | 决策码 | 系统输出的最终决策类型编码，如 AUTO_SINGLE_SHIP、SPLIT_SHIP 等 |
| decision_reason | 决策原因 | 白盒解释：为什么选了这个方案 |
| system_action | 系统动作 | 决策后系统自动执行的动作，如锁库、下发 WMS、通知承运商 |
| manual_action | 人工建议 | 需要人工介入时的建议操作 |
| customer_impact | 客户影响 | 该决策对客户体验的影响说明 |
| trace | 决策链路追踪 | 完整的决策过程日志，包含每一步的输入、输出、耗时 |
| request_idempotency_key | 请求幂等键 | 用于保证同一请求重复调用时返回相同结果，防止重复锁库 |
| order_version | 订单版本号 | 乐观锁版本号，防止并发修改导致的数据不一致 |

### 1.7 一页读懂 5 个关键策略字段

| 策略字段 | 中文含义 | 取值 | 说明 |
|----------|----------|------|------|
| customer_experience_penalty | 客户体验惩罚系数 | 0.0 ~ 1.0 | 值越高，系统越倾向于选择 ETA 更短的方案，即使成本更高。用于 VIP 客户或高优先级订单 |
| gift_same_package_required | 赠品必须同包 | true / false | 为 true 时，赠品 SKU 必须与主商品放在同一个包裹中，不可拆开发货 |
| bundle_same_warehouse_required | 组合商品必须同仓 | true / false | 为 true 时，组合商品（bundle）的所有子 SKU 必须从同一个仓库发出 |
| partial_fulfillment_allowed | 允许部分履约 | true / false | 为 true 时，库存不足的 SKU 可以先发有货部分，缺货部分挂起等补货；为 false 时，任一 SKU 缺货则整单挂起 |
| high_value_force_single_delivery | 高价值订单强制单包 | true / false | 为 true 时，订单金额超过阈值的订单强制从单一仓库、单一包裹发出，降低丢件和理赔风险 |

### 1.8 一页读懂关键公式

#### 1.8.1 候选仓局部评分

> 人话：对每个候选仓打一个综合分，分越低越好（成本越低、时效越快、风险越小）。

```
Score_w = α × NormCost_w + β × NormETA_w + γ × NormRisk_w + δ × CapacityPenalty_w
```

- `NormCost_w`：该仓方案的归一化成本（0~1）
- `NormETA_w`：该仓方案的归一化 ETA（0~1）
- `NormRisk_w`：该仓方案的归一化风险分（0~1）
- `CapacityPenalty_w`：该仓的容量惩罚分
- `α, β, γ, δ`：权重系数，由算法参数表配置，总和为 1

#### 1.8.2 ETA 公式

> 人话：送达时间 = 仓内排队等待 + 仓内处理 + 运输时间，再根据历史准点率做风险校准。

```
ETA = T_queue + T_process + T_transit
ETA_calibrated = ETA × (1 + RiskFactor)
```

- `T_queue`：仓内排队时间，取决于当前仓的待处理订单量和产能
- `T_process`：仓内处理时间（拣货、打包、出库），取决于 SKU 数量和仓效率
- `T_transit`：运输时间，取决于承运商、起止地、运输方式
- `RiskFactor`：风险校准因子，基于该线路历史准点率计算。准点率越低，RiskFactor 越大

#### 1.8.3 计费重公式

> 人话：快递费按"实际重量"和"体积换算重量"中较大的那个收费。

```
BillingWeight = MAX(ActualWeight, VolumetricWeight)
VolumetricWeight = (Length × Width × Height) / DimFactor
```

- `ActualWeight`：包裹实际称重（kg）
- `VolumetricWeight`：体积重，用长宽高除以体积因子换算
- `DimFactor`：体积因子，不同承运商不同（通常快递 6000，物流 5000）

#### 1.8.4 包裹运费公式

> 人话：运费 = 首重费 + 续重费，续重按阶梯价计算。

```
Freight = FirstWeightPrice + ContinuedWeightPrice × (BillingWeight - FirstWeight)
```

- 当 `BillingWeight ≤ FirstWeight` 时，`Freight = FirstWeightPrice`
- 续重部分可能有阶梯价：不同重量段单价不同，需要分段累加
- 偏远地区、特殊品类可能有附加费

#### 1.8.5 综合成本公式

> 人话：一个方案的总成本 = 所有包裹的（运费 + 包材 + 操作费）+ 拆单惩罚 + 调拨成本 + 容量惩罚 + 库存机会成本。

```
TotalCost = Σ(Freight_i + PackagingCost_i + OpsCost_i) + SplitPenalty + TransferCost + CapacityPenalty + InventoryOpportunityCost
```

- `i`：方案中的第 i 个包裹
- `SplitPenalty`：拆单数量超过 1 时触发，= 拆单惩罚系数 × (包裹数 - 1)
- `TransferCost`：仅在调拨合发方案中存在
- `CapacityPenalty`：仓负荷超过阈值时触发
- `InventoryOpportunityCost`：基于 SKU 周转率和剩余库存深度计算

#### 1.8.6 最终评分公式

> 人话：最终方案评分 = 成本分 + 时效分 + 风险分 + 客户体验惩罚，分越低越好。

```
FinalScore = α × NormCost + β × NormETA + γ × NormRisk + CustomerExperiencePenalty
```

- 所有候选方案按 `FinalScore` 升序排列，取最优
- `CustomerExperiencePenalty`：当方案不满足客户体验策略时的额外惩罚分
- 如果最优方案的 `FinalScore` 超过阈值，触发人工审核

#### 1.8.7 毛利公式

> 人话：毛利 = 订单收入 − 商品成本 − 履约成本。毛利率低于红线时触发财务熔断。

```
GrossProfit = OrderRevenue - COGS - TotalFulfillmentCost
GrossMarginRate = GrossProfit / OrderRevenue × 100%
```

- `OrderRevenue`：订单实付金额（含运费收入）
- `COGS`：商品成本（Cost of Goods Sold）
- `TotalFulfillmentCost`：总履约成本（= TotalCost 公式的结果）
- 当 `GrossMarginRate < 阈值` 时，触发财务熔断，阻断自动下发，等待人工审批

---

当前履约决策存在以下结构性问题：

1. 规则分散。库存、仓、承运商、平台约束、财务阈值、人工经验分散在多个系统中，没有统一决策口径。
2. 结果黑盒。业务看得到"分到了哪个仓"，看不到"为什么是它、为什么不是别的仓、为什么失败"。
3. 库存不一致。高并发下常见"求解时有货，执行时无货"，导致锁库失败、重复重试、错分仓。
4. 装箱后置。很多场景先算仓，再到打单阶段才发现超长超重、禁混、温区冲突，导致理论方案无法执行。
5. ETA 过度乐观。仅依赖承运商 ETA 或平均值，缺少截单时间、班次、排队、历史准点率、天气停发等因素。
6. 异常缺少统一出口。库存缺货、地址异常、承运商失败、余额不足、取消改单等场景没有标准化最终决策和处理建议。

因此，本产品不是新增几条规则，而是重构整个分仓决策链路。

## 决策码表

每一次分仓决策必须输出以下决策码之一，不允许出现未分类的结果。

| 序号 | decision_code | 中文含义 | 触发条件 | system_action | manual_action |
|------|---------------|----------|----------|---------------|---------------|
| 1 | AUTO_SINGLE_SHIP | 自动单仓直发 | 单一仓库可满足全部 SKU，成本与时效最优 | 锁库 → 下发 WMS → 通知承运商 | 无 |
| 2 | SPLIT_SHIP | 自动拆单发货 | 无单仓可满足全部 SKU，拆单方案成本最优且包裹数 ≤ max_package_count | 按包裹分别锁库 → 分别下发 WMS | 无 |
| 3 | TRANSFER_CONSOLIDATE | 调拨合发 | 调拨合发方案的综合成本低于拆单方案，且调拨时效可接受 | 创建调拨单 → 目标仓收货后锁库 → 下发 WMS | 无 |
| 4 | PARTIAL_FULFILL | 部分履约 | partial_fulfillment_allowed = true，部分 SKU 有货，部分缺货 | 有货部分锁库下发，缺货部分创建挂起单 | 通知客户部分发货，跟进缺货 SKU 补货 |
| 5 | PENDING_INVENTORY | 挂起等补货 | 全部或关键 SKU 缺货，预计补货时间在可接受范围内 | 创建挂起单，设置补货监听 | 跟进补货进度，超时未补则升级 |
| 6 | PENDING_REVIEW | 挂起等审核 | 财务熔断触发、高风险订单、规则冲突需人工裁决 | 创建审核工单，暂停自动流程 | 审核通过后手动放行或取消 |
| 7 | FORCE_SINGLE_WAREHOUSE | 强制单仓 | high_value_force_single_delivery = true 且订单金额超阈值 | 仅从最优单仓锁库下发，不拆单 | 无 |
| 8 | DOWNGRADE_CARRIER | 降级承运商 | 首选承运商不可用（余额不足、停发、超负荷），备选承运商可用 | 切换承运商 → 重新计算运费 → 锁库下发 | 通知运营承运商降级 |
| 9 | DOWNGRADE_SLA | 降级时效 | 所有方案均无法满足承诺时效，但客户可接受延迟 | 按最优成本方案执行，更新客户 ETA | 通知客户时效变更 |
| 10 | BLOCK_ADDRESS | 地址异常阻断 | 收货地址无法解析、在黑名单中、或无承运商覆盖 | 阻断下发，创建地址异常工单 | 联系客户确认地址 |
| 11 | BLOCK_RISK | 风控阻断 | 订单命中风控规则（欺诈、刷单、异常地址） | 阻断下发，创建风控审核工单 | 风控团队审核 |
| 12 | BLOCK_FINANCE | 财务阻断 | 毛利率低于红线或承运商余额不足 | 阻断下发，创建财务审核工单 | 财务审批通过后放行 |
| 13 | BLOCK_COMPLIANCE | 合规阻断 | SKU 涉及禁售、限售、资质缺失 | 阻断下发，创建合规审核工单 | 合规团队审核 |
| 14 | CANCEL_REFUND | 取消退款 | 客户主动取消且订单未下发 WMS | 释放库存预占，发起退款 | 无 |
| 15 | INTERCEPT_WMS | 拦截 WMS | 客户取消或改单，但订单已下发 WMS 且未出库 | 向 WMS 发送拦截指令 | 确认拦截结果，失败则转逆向 |
| 16 | REVERSE_RETURN_FLOW | 逆向退货流程 | 订单已出库或已签收，需要退货退款 | 创建逆向单，通知承运商取件或客户寄回 | 跟进退货物流，收货后退款 |

## 决策解释必须包含的 7 个字段

每一次决策输出必须包含以下 7 个字段，缺一不可：

| 字段 | 中文含义 | 说明 |
|------|----------|------|
| decision_code | 决策码 | 上表中的 16 种之一 |
| decision_reason | 决策原因 | 白盒解释，说明为什么选择了这个方案。必须包含：选中仓/方案的关键指标、落选方案的淘汰原因 |
| system_action | 系统动作 | 决策后系统自动执行的动作列表 |
| manual_action | 人工建议 | 需要人工介入时的建议操作。如果无需人工介入，填"无" |
| customer_impact | 客户影响 | 该决策对客户体验的影响。如"正常时效""时效延迟约 1 天""部分商品缺货" |
| trace | 决策链路 | 完整的决策过程追踪 ID，可关联查询每一步的详细日志 |
| confidence | 置信度 | 系统对该决策的置信度（0~1）。低于阈值时建议人工复核 |

## 规则优先级

当多条规则同时生效且存在冲突时，按以下优先级从高到低裁决：

| 优先级 | 规则类别 | 说明 |
|--------|----------|------|
| 1 | 合规与法律 | 禁售、限售、资质要求、跨境法规。不可违反，不可降级 |
| 2 | 安全与风控 | 危险品管控、欺诈拦截、黑名单。不可违反，不可降级 |
| 3 | 物理可履约 | 库存可用（ATP > 0）、温区兼容、尺寸可装箱、承运商可达。不满足则方案不合法 |
| 4 | 库存一致性 | 预占成功、版本一致、无并发冲突。不满足则重试或回滚 |
| 5 | 客户 SLA | 承诺时效、承诺送达日。优先保障，但可在客户同意下降级 |
| 6 | 财务红线 | 毛利率底线、承运商余额。触发熔断，等待审批 |
| 7 | 客户体验策略 | 赠品同包、组合同仓、高价值单包等。尽量满足，冲突时可降级 |
| 8 | 成本最优 | 在满足以上所有约束后，选择总成本最低的方案 |
| 9 | 仓网均衡 | 在成本相近时，优先选择负荷较低的仓，避免局部过载 |
| 10 | 承运商偏好 | 在其他条件相同时，优先选择历史准点率高、服务质量好的承运商 |

## 不可违反的 6 条红线规则

以下规则在任何场景下都不可违反，不可通过人工 Override 绕过：

| 序号 | 红线规则 | 说明 |
|------|----------|------|
| 1 | 禁售商品不可发货 | 命中禁售名单的 SKU，无论库存是否充足，一律阻断 |
| 2 | 危险品不可混装普通品 | hazmat_type 非空的 SKU 不可与普通 SKU 放入同一包裹 |
| 3 | 温区不兼容不可同包 | 冷冻与常温 SKU 不可放入同一包裹 |
| 4 | ATP ≤ 0 不可锁库 | 可用库存为零或负数时，禁止执行预占操作 |
| 5 | 未通过风控不可下发 | 命中风控规则的订单，必须等待风控审核通过后才能下发 |
| 6 | 幂等键冲突不可重复执行 | 相同 request_idempotency_key 的请求，直接返回上次结果，不重复锁库 |
