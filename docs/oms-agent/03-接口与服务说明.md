## 1. 核心 RPC 接口（可直接给 Skill 调用）

### 1.1 POST /dispatch/sales-order
- 服务：DispatchService.salesOrderDispatch
- 输入：SalesOrderDispatchParam
  - tenantId, channelId, merchantNo, referenceNo, orderNo
  - items: List<OrderItemDO>（SKU/数量/物理属性）
  - carrierSCAC, carrierName, shippingService
  - zipcode, country, address1/2/3, city, state
  - defaultRules: List<Integer>（规则 ID 列表）
  - customRules: List<CustomRuleDTO>（自定义规则）
  - warehouseFilterStrategies, dispatchStrategies, backupDispatchStrategy, deliveryGenerateStrategy
- 输出：SalesOrderDispatchRespDTO
  - status: Integer（DispatchStatusEnum: 1=SUCCESS, 2=NO_WAREHOUSE, -1=EXCEPTION）
  - dispatchList: List<DispatchDTO>（每个拆单结果含仓库/承运商/商品行）
  - logs: List<String>（过程日志）
  - exceptionMsg: String
  - eventId: String
- 复用建议：**直接给 warehouse_allocation skill 调用**，是分仓的核心入口

### 1.2 POST /dispatch/purchase-order
- 服务：DispatchService.purchaseOrderDispatch
- 输入：PurchaseOrderDispatchParam
- 输出：PurchaseOrderDispatchRespDTO
- 复用建议：partial，采购场景专用

### 1.3 POST /dispatch/delivery-order
- 服务：DispatchService.deliveryOrderDispatch
- 输入：DeliveryOrderDispatchParam
- 输出：DeliveryOrderDispatchRespDTO
- 复用建议：partial，出库场景专用

### 1.4 POST /dispatch/work-order
- 服务：DispatchService.workOrderDispatch
- 输入：WorkOrderDispatchParam
- 输出：WorkOrderDispatchRespDTO
- 复用建议：partial，工单场景专用

### 1.5 GET /dispatch-log/{eventId}
- 服务：DispatchLogService.queryDispatchLogs
- 输入：eventId (String)
- 输出：List<DispatchLogVO>（含策略列表/拆单结果/状态/过程日志/异常信息）
- 复用建议：**直接给 oms_query / order_analysis skill 调用**

### 1.6 GET /channel?channelNo=xxx
- 服务：ChannelApi (Feign 远程调用)
- 输入：channelNo
- 输出：List<ChannelDTO>
- 复用建议：**共享能力**，多个 skill 可能需要渠道信息

## 2. Mapping RPC 接口（可直接给 Skill 调用）

### 2.1 POST /mapping/shipping/execute
- 服务：ShippingMappingExecuteService.applyShippingMappingRules
- 输入：channelId (Long) + inputConditions (Map<String,String>)
- 输出：Map<String,String>（映射后的输出条件）
- 复用建议：**直接给 shipping_rate skill 调用**，承运商/服务自动推荐的核心

### 2.2 POST /mapping/condition/execute
- 服务：ConditionMappingDOService（执行条件映射）
- 输入：ExecuteConditionMappingDTO
- 输出：ExecuteConditionMappingResultDTO
- 复用建议：**直接给 shipping_rate / warehouse_allocation skill 调用**

### 2.3 POST /mapping/add-or-update
- 服务：OneToOneMappingDOService.addOrUpdateMapping
- 输入：OneToOneMappingDTO（mappingType/mappedType/originValue/mappedValue/mappingKey/merchantNo）
- 输出：Long (ID)
- 复用建议：配置管理接口，非 skill 运行时调用

### 2.4 POST /mapping/batch
- 服务：OneToOneMappingDOService.batchOneToOneMapping
- 输入：OneToOneMappingBatchDTO
- 输出：OneToOneMappingBatchDTO
- 复用建议：批量配置接口

### 2.5 GET /mapping/list
- 服务：OneToOneMappingDOService.getOneToOneMappingList
- 输入：OneToOneMappingQueryDTO
- 输出：List<OneToOneMappingDTO>
- 复用建议：**可给 shipping_rate skill 查询映射关系**

## 5. 核心 Service 逻辑说明

### 5.1 DispatchService
- 核心分仓拆单服务
- 执行链路：构建 OrderContext → 执行 WarehouseFilterStrategy 过滤 → 执行 ItemLineDispatchStrategy 分配 → 执行 BackupDispatchStrategy 兜底 → 执行 DeliveryGenerateStrategy 生成出库单 → 持久化日志
- 支持 4 种订单类型，但核心逻辑在 salesOrderDispatch

### 5.2 ShippingMappingExecuteService
- 独立的规则执行引擎
- 输入：channelId + 条件 Map → 按优先级匹配规则 → 输出映射结果 Map
- 与 DispatchService 解耦，可独立调用

### 5.3 BaseStrategy（策略基类）
- checkWarehouseFulfillment：逐 SKU 校验单仓库存是否满足需求
- checkRoutingResultFulfillment：校验路由结果是否覆盖全部订单行
- 这两个方法是所有策略的公共依赖

## 6. 关键区分：可复用 vs 仅供展示

| 类型 | 接口 | 建议 |
|------|------|------|
| 可直接给 skill 调用 | /dispatch/sales-order, /dispatch-log/{eventId}, /mapping/shipping/execute, /mapping/condition/execute, /mapping/list, /channel | 后端 RPC 接口，有明确输入输出 |
| 仅供前端展示/配置 | /mapping/single/*, /mapping/multi-conditions/*, /mapping/multiple/rule/* | CRUD + 下拉选项，skill 不应直接调用 |
| 不建议调用 | /admin-api/invoke | 安全风险 |

## 7. 补充发现（2026-04-08 深度扫描）

### 7.1 Dispatch 响应的完整数据结构（关键发现）

之前只关注了查询接口，忽略了 dispatch 执行接口的响应本身就包含完整数据。

`POST /dispatch/sales-order` 返回的 `SalesOrderDispatchRespDTO` 结构：

```
SalesOrderDispatchRespDTO extends CommonDispatchRespDTO:
  status: Integer              # 1=成功, 2=无仓库, -1=异常
  logs: List<String>           # 过程日志（逐步记录）
  exceptionMsg: String         # 异常信息
  exceptionType: String        # 异常类型
  eventId: String              # 事件 ID（反查链路关键）
  dispatchList: List<DispatchDTO>  # 完整分仓结果
```

每个 `DispatchDTO` 包含：
```
DispatchDTO:
  dispatchNo: String                    # 拆单编号
  warehouseId: String                   # 仓库 ID
  warehouseAccountingCode: String       # 仓库 Accounting Code
  warehouseName: String                 # 仓库名称
  carrierScac: String                   # 承运商 SCAC
  carrierName: String                   # 承运商名称
  deliveryService: String               # 配送服务
  shipMethod: String                    # 发货方式
  dispatchItemLineList: List<DispatchItemLineDTO>  # 商品行列表
```

每个 `DispatchItemLineDTO` 包含：
```
DispatchItemLineDTO:
  dispatchNo: String      # 拆单编号
  orderItemId: Long       # 订单商品行 ID
  itemLineNo: String      # 行号
  sku: String             # SKU
  uom: String             # 计量单位
  qty: Integer            # 数量
```

### 7.2 对 oms_query 的意义

这意味着 dispatch 接口的响应已经包含了 oms_query 需要的大部分全景数据：
- 分仓结果（仓库/承运商/商品行）→ dispatchList
- 过程日志 → logs
- 异常信息 → exceptionMsg + exceptionType
- 事件 ID → eventId（反查链路）
- 状态 → status

问题是：dispatch 是"执行"接口，不是"查询"接口。每次调用都会重新执行分仓逻辑。
解决方案：需要一个"只查不执行"的接口，返回历史 dispatch 结果。

### 7.3 ProductDTO 完整字段（SKU 主数据）

ProductDTO 包含的字段远比之前认知的丰富：
- 物理属性：length / width / height / weight / uomDimensions / uomWeight
- 承运商关联：carrierScac / carrierName / deliveryName
- 品类：category / subCategory
- 组合商品：productComboVOS（List<ProductComboDTO>）
- 标签：tags / productTags
- 序列号/批次号：serialProduct / lotNumberProduct

这些数据在 OrderContext.skuProductMap 中被加载，可以支撑 cartonization 和 oms_query 的 SKU 上下文。

### 7.4 WarehouseDTO 完整字段

WarehouseDTO 包含完整的仓库信息：
- 基础：id / facilityCode / facilityName / accountingCode / type / status
- 地址：country / state / city / zipCode / address1 / address2 / lat / lng
- 配置：timeZone / zipcodeRange / warehouseSort / channelNo / channelName
- 扩展：warehouseExtData (JSONObject)

### 7.5 距离服务（独立外部服务）

存在一个独立的距离计算服务：
- 认证：PassloginVO（username/password/channel/platform/device）
- 配置：DistanceServiceProperties（loginUrl/baseUrl）+ WareHouseDistanceConfig（distanceUrl/timeout/maxRetryAttempts）
- 返回：OrderDispatchDistanceVO（origin/destination/distance）
- 存储：WareHouseDistanceDO（warehouse_distance 表）

这个服务可以支撑 eta / cost / warehouse_allocation 的距离计算需求。

### 7.6 ChannelDTO 完整字段

ChannelDTO 包含渠道和连接器信息：
- 渠道：channelId / channelNo / channelName / merchantId
- 连接器：connectorDTO（connectorName / connectorCompany / connectorApiUrl / connectorCountry）
- 状态：orderDownloadStatus / inventoryUploadStatus / connectionStatus / authStatus
