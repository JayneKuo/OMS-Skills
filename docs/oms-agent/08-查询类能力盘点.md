## 一、现有查询类 API 清单

### 1. 订单 / 拆单查询

| API | 方法 | 业务含义 | 返回结构 | 可给 skill 用 |
|-----|------|---------|---------|-------------|
| `/dispatch-log/{eventId}` | GET | 按 eventId 查询拆单日志 | List<DispatchLogVO>（摘要：orderNo/eventId/eventType/summary/status/eventTime） | yes → oms_query |

DispatchLogVO 字段：merchantNo, orderNo, eventId, eventType, summary, status, eventTime

注意：这是摘要级查询，返回的是轻量 VO，不是完整的 DispatchLogDO（含策略/过程日志/拆单结果）。

### 2. 映射查询

| API | 方法 | 业务含义 | 返回结构 | 可给 skill 用 |
|-----|------|---------|---------|-------------|
| `/mapping/list` | GET | 查询一对一映射列表 | List<OneToOneMappingDTO> | yes → shipping_rate |
| `/mapping/id` | GET | 按 ID 查询一对一映射 | OneToOneMappingDTO | yes → shipping_rate |
| `/mapping` | POST | 分页查询一对一映射 | RespPage<OneToOneMappingDTO> | partial（前端分页） |
| `/mapping/condition` | GET | 查询所有条件映射 | List<ConditionMappingDTO> | yes → shipping_rate |
| `/mapping/condition/id` | GET | 按 ID 查询条件映射 | ConditionMappingDTO | yes → shipping_rate |
| `/mapping/multiple/rule/page` | GET | 分页查询 Shipping Mapping 规则 | RespPage<ShippingMappingRuleDTO> | partial（前端分页） |

### 3. 渠道查询

| API | 方法 | 业务含义 | 返回结构 | 可给 skill 用 |
|-----|------|---------|---------|-------------|
| `/channel?channelNo=xxx` | GET | 查询渠道信息 | List<ChannelDTO> | yes → 共享能力 |

### 4. 映射执行（查询+计算混合）

| API | 方法 | 业务含义 | 返回结构 | 可给 skill 用 |
|-----|------|---------|---------|-------------|
| `/mapping/condition/execute` | POST | 执行条件映射规则 | ExecuteConditionMappingResultDTO | yes → shipping_rate |
| `/mapping/condition/multi` | POST | 执行 Shipping Mapping 规则 | Map<String,String> | yes → shipping_rate |

### 5. 认证

| API | 方法 | 业务含义 | 返回结构 | 可给 skill 用 |
|-----|------|---------|---------|-------------|
| `/iam/token` | POST | 获取 access_token | IamTokenRespDTO | yes → 共享能力 |
| `/iam/user-info` | GET | 获取用户信息 | UserInfoRespDTO | yes → 共享能力 |

## 二、按业务能力归类

### Q1. 订单拆单结果查询
- 接口：`GET /dispatch-log/{eventId}`
- 返回：摘要级（orderNo, eventId, eventType, summary, status, eventTime）
- 适合 skill：oms_query（核心）
- 局限：当前返回的是 DispatchLogVO 摘要，不含完整的策略列表、过程日志、拆单明细
- 完整数据在 DispatchLogDO 中（含 filterStrategies, dispatchStrategies, dispatchLogs, dispatchList），但当前 API 只返回摘要

### Q2. 一对一映射查询
- 接口：`GET /mapping/list`, `GET /mapping/id`
- 返回：OneToOneMappingDTO（mappingType, mappedType, originValue, mappedValue, mappingKey, status）
- 适合 skill：shipping_rate（查询 SKU/承运商/服务映射关系）
- 也可作为 oms_query 的补充（查询 SKU 映射）

### Q3. 条件映射查询
- 接口：`GET /mapping/condition`, `GET /mapping/condition/id`
- 返回：ConditionMappingDTO（conditionInList, conditionOut, conditionOutValue, mappingKey）
- 适合 skill：shipping_rate

### Q4. Shipping Mapping 规则查询
- 接口：`GET /mapping/multiple/rule/page`
- 返回：ShippingMappingRuleDTO（channelNo, ruleName, priority, status, conditions, outputs）
- 适合 skill：shipping_rate

### Q5. 渠道信息查询
- 接口：`GET /channel?channelNo=xxx`
- 返回：ChannelDTO
- 适合 skill：共享能力（warehouse_allocation, shipping_rate 都需要）

### Q6. 条件映射执行
- 接口：`POST /mapping/condition/execute`
- 返回：ExecuteConditionMappingResultDTO
- 适合 skill：shipping_rate（不是纯查询，是查询+规则匹配）

### Q7. Shipping Mapping 规则执行
- 接口：`POST /mapping/condition/multi`
- 输入：MultiplyMappingDTO（channelNo, merchantNo, inputConditions）
- 返回：Map<String,String>
- 适合 skill：shipping_rate（不是纯查询，是查询+规则匹配）

### Q8. 认证 / 用户信息查询
- 接口：`POST /iam/token`, `GET /iam/user-info`
- 适合 skill：共享能力（所有 skill 的前置依赖）

## 五、补充发现（2026-04-08 深度扫描）

### 关键发现：Dispatch 响应本身就是最丰富的数据源

之前的盘点只关注了"查询接口"，忽略了 dispatch 执行接口的响应。

`POST /dispatch/sales-order` 的返回 `SalesOrderDispatchRespDTO` 包含：
- `dispatchList`：完整分仓结果（仓库/承运商/服务/商品行明细）
- `logs`：过程日志（逐步操作记录）
- `exceptionMsg` + `exceptionType`：异常信息
- `eventId`：事件 ID（反查链路关键）
- `status`：拆单状态

这意味着：
1. 每次 dispatch 执行后，响应中已经包含了 oms_query 需要的大部分全景数据
2. 如果能把 dispatch 响应持久化并提供"只查不执行"的查询入口，oms_query 就能获取完整数据
3. 当前 `GET /dispatch-log/{eventId}` 只返回 DispatchLogVO 摘要，但底层 DispatchLogDO 存储了完整数据

### 新增查询能力评估

| 查询能力 | 数据来源 | 当前可获取方式 | 支持度提升 |
|---------|---------|-------------|-----------|
| 分仓结果明细 | DispatchDTO in SalesOrderDispatchRespDTO | dispatch 响应 / DispatchLogDO | ⚠️→✅（如果扩展 dispatch-log API） |
| 过程日志 | CommonDispatchRespDTO.logs | dispatch 响应 / DispatchLogDO.dispatchLogs | ⚠️→✅（同上） |
| 异常详情 | CommonDispatchRespDTO.exceptionMsg | dispatch 响应 / DispatchLogDO | ⚠️→✅（同上） |
| SKU 主数据 | ProductDTO（含物理属性+承运商+品类+组合商品） | OrderContext.skuProductMap（运行时） | ⚠️ 仍需独立查询 API |
| 仓库完整信息 | WarehouseDTO（含地址/坐标/邮编范围/时区） | OrderContext.warehouseDataMap（运行时） | ⚠️ 仍需独立查询 API |
| 仓库距离 | OrderDispatchDistanceVO + 独立距离服务 | WareHouseDistanceDO + DistanceServiceProperties | ⚠️→✅（有独立服务可调用） |
| 渠道+连接器信息 | ChannelDTO + ConnectorDTO | GET /channel?channelNo=xxx | ✅ 已有 |
| 组合商品 | ProductComboDTO | ProductDTO.productComboVOS | ⚠️ 需通过 SKU 查询获取 |

### 结论：oms_query 落地的最小后端改动

只需要 1 个改动就能大幅提升 oms_query 的能力：

**扩展 `GET /dispatch-log/{eventId}` 返回完整 DispatchLogDTO（而非 DispatchLogVO 摘要）**

因为 DispatchLogDO 中已经存储了：
- dispatchList（完整分仓结果）
- dispatchLogs（过程日志）
- filterStrategies / dispatchStrategies / backupDispatchStrategy（策略命中）
- exceptionMsg（异常信息）
- customStrategies / dispatchConfig（自定义策略和配置）

这些数据已经在数据库里了，只是当前 API 只返回了摘要 VO。扩展返回结构是最小改动、最大收益的方案。

## 六、重大发现：History/Phase 阶段采集框架（2026-04-08）

### 框架概述

OMS 代码中存在一套完整的方法级阶段采集框架（linker-framework/linker-common）：

- `@HistoryMethod`：标注在业务方法上，声明 businessType / phase / description / channel
- `@HistoryKey`：标注在关键字段上（如 OrderItemDO.referenceNo），用于提取订单号等关键标识
- `HistoryAspect`：AOP 切面，自动拦截标注了 @HistoryMethod 的方法，采集请求参数和响应结果
- `PhaseEvent`：阶段事件数据结构，包含 businessType / phase / methodName / phaseRequestData / phaseResponseData / channel / source / executeStatus / errorMessage / keyValue / createDateTime
- `PhaseHandler`：阶段事件处理器接口，负责持久化

### 对 oms_query 的意义

这套框架意味着：

1. OMS 系统可能已经在自动记录每个业务阶段的执行历史（routing history）
2. 每次 dispatch 流程中的各步骤（过滤/分配/兜底/出库单生成）如果标注了 @HistoryMethod，其输入输出和执行状态都会被自动采集
3. PhaseEvent 中的 keyValue（从 @HistoryKey 提取的 referenceNo/orderNo）可以用于按订单号反查所有阶段历史
4. 这正是 oms_query 需要的"关键事件时间线"和"异常上下文"数据源

### 对查询能力的提升

| 查询能力 | 之前评估 | 有了 History 框架后 |
|---------|---------|-------------------|
| 按 orderNo 查询历史 | ❌ 不支持 | ✅ 可能支持（通过 keyValue = orderNo 查询 PhaseEvent） |
| 关键事件时间线 | ⚠️ 只有 dispatch log 摘要 | ✅ 可能有完整的阶段级时间线 |
| 异常上下文 | ⚠️ 只有 exceptionMsg | ✅ PhaseEvent 包含 errorMessage + phaseRequestData + phaseResponseData |
| 每步操作的输入输出 | ⚠️ 只有 logs 列表 | ✅ PhaseEvent 包含 phaseRequestData / phaseResponseData |

### 待确认

- PhaseHandler 的具体实现在哪里？持久化到什么存储？
- staging 环境是否有对应的 PhaseEvent 查询 API？
- dispatch 流程的各步骤是否已标注 @HistoryMethod？

如果以上都是肯定的，oms_query 的能力将大幅提升——可以按 orderNo 查询完整的 routing history，包括每个阶段的输入输出、执行状态、异常信息和时间线。
