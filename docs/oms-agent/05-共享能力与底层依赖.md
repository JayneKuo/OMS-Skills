## 2. 规划外但有价值的隐藏能力

### 2.1 规则命中记录（Rule Hit Logging）
- 来源：DispatchLogDO 中的 filterStrategies / dispatchStrategies / backupDispatchStrategy / customStrategies
- 价值：每次分仓都持久化了完整的规则命中链路，可用于：
  - order_analysis 做"为什么分到这个仓"的白盒解释
  - 规则命中率统计
  - 规则冲突检测
- 建议归属：order_analysis 的数据源 + 共享分析能力

### 2.2 自定义规则条件组合引擎
- 来源：CustomRuleDTO（渠道 + 商品数量 + 承运商 + 服务，四维条件组合，每个条件有操作符和启用开关）
- 价值：已经是一个规则引擎雏形，可扩展为通用规则评估能力
- 建议归属：warehouse_allocation 的底层依赖

### 2.3 Shipping Mapping 规则执行引擎
- 来源：ShippingMappingExecuteService（channelId + 条件 Map → 按优先级匹配 → 输出 Map）
- 价值：独立的规则执行引擎，与 DispatchService 解耦
- 建议归属：shipping_rate 的核心依赖，也可作为通用条件映射执行能力

### 2.4 仓库排序 / 优先级机制
- 来源：OrderContext.warehouseSort + WarehouseData.sort / points
- 价值：多仓候选时的优先级排序基础，可扩展为多维评分模型
- 建议归属：warehouse_allocation 的底层依赖

### 2.5 SKU-UOM 复合键处理
- 来源：ItemLineUomSkuWrapper + SkuUomWrapper + OrderContext.itemLineSkuMapping
- 价值：同一 SKU 不同计量单位的标准化处理
- 建议归属：共享数据处理能力，cartonization / oms_query / warehouse_allocation 都需要

### 2.6 OrderItemDO 物理属性字段
- 来源：OrderItemDO（length/width/height/weight/fragile/stackable/containBattery/category）
- 价值：OMS 中已有的商品物理属性，可直接桥接到装箱引擎
- 建议归属：cartonization 的数据源 + 共享数据能力

### 2.7 合单标记
- 来源：DispatchStrategyEnum.MERGE_ORDER (ruleType=7)
- 价值：合单概念已存在，可扩展为订单聚合/蓄水池能力
- 建议归属：warehouse_allocation 的扩展能力

### 2.8 订单类型标记
- 来源：DispatchStrategyEnum 中的 CONFIG_SAMPLE_ORDER / CONFIG_CROSS_BORDER_ORDER / CONFIG_REPLACEMENT_ORDER
- 价值：不同订单类型影响分仓/装箱/运费策略选择，可作为 Agent 任务路由依据
- 建议归属：共享配置能力

### 2.9 拆单过程日志
- 来源：OrderContext.logs → DispatchLogDO.dispatchLogs
- 价值：逐步操作记录，是白盒解释的数据基础
- 建议归属：order_analysis 的核心数据源

### 2.10 动态 Bean 调用
- 来源：OasAdminController.invokeBeanMethod
- 价值：运维/调试能力，Agent 不应直接使用
- 建议归属：不纳入 skill 体系，仅作为运维工具

## 4. 查询类共享能力候选（2026-04-07 补充）

以下查询能力在代码中隐式存在但未暴露为独立 API，建议作为共享查询能力候选：

| # | 能力 | 当前状态 | 适合支持的 skill | 建议 |
|---|------|---------|----------------|------|
| QS1 | 认证服务 (POST /iam/token) | 已有 API | 全部 skill | 共享能力层，所有 skill 的前置依赖 |
| QS2 | 渠道查询 (GET /channel) | 已有 API | warehouse_allocation, shipping_rate | 共享能力层 |
| QS3 | 仓库库存查询 | 隐式（OrderContext.warehouseInventory） | warehouse_allocation | 底层依赖，后续建议暴露为独立 API |
| QS4 | SKU 主数据查询 | 隐式（OrderContext.skuProductMap） | cartonization, oms_query | 底层依赖，后续建议暴露为独立 API |
| QS5 | 仓库距离查询 | 隐式（WareHouseDistanceDO） | eta, cost, warehouse_allocation | 底层依赖，后续建议暴露为独立 API |
| QS6 | 订单商品行查询 | 隐式（OrderItemDO，嵌在 dispatch 请求中） | oms_query, cartonization | 底层依赖，后续建议暴露为独立 API |
| QS7 | 自定义规则查询 | 隐式（OrderDispatchCustomRuleDO） | warehouse_allocation | 底层依赖 |
| QS8 | 默认规则查询 | 隐式（OrderDispatchDefaultRuleDO） | warehouse_allocation | 底层依赖 |

说明：
- QS1/QS2 已有 API，直接作为共享能力
- QS3-QS8 在代码逻辑中存在但无独立查询 API，当前只能通过 dispatch 流程间接获取
- 不建议将这些能力提升为独立顶层 skill，应保持为共享能力或底层依赖
- 后续如果需要，可以建议后端团队为 QS3-QS6 暴露独立查询 API

## 5. 补充发现的底层依赖能力（2026-04-08 深度扫描）

### 5.1 距离服务（独立外部服务）

代码中存在一个独立的距离计算服务，有完整的认证和调用链路：

| 组件 | 说明 |
|------|------|
| PassloginVO | 认证参数（username/password/channel/platform/device） |
| DistanceServiceProperties | 服务配置（loginUrl/baseUrl） |
| WareHouseDistanceConfig | 距离配置（distanceUrl/timeout/maxRetryAttempts） |
| OrderDispatchDistanceVO | 返回结构（origin/destination/distance） |
| WareHouseDistanceDO | 持久化表（warehouse_distance） |

适合支持的 skill：eta / cost / warehouse_allocation
建议归属：共享能力层（距离计算服务）

### 5.2 SKU 主数据完整结构（ProductDTO）

ProductDTO 比之前认知的丰富得多：
- 物理属性：length / width / height / weight / uomDimensions / uomWeight
- 承运商关联：carrierScac / carrierName / deliveryName
- 品类：category / subCategory
- 组合商品：productComboVOS
- 标签：tags / productTags
- 序列号/批次号：serialProduct / lotNumberProduct

适合支持的 skill：cartonization（物理属性）/ oms_query（SKU 上下文）/ shipping_rate（承运商关联）
建议归属：共享数据能力（SKU 主数据查询）

### 5.3 仓库完整结构（WarehouseDTO）

WarehouseDTO 包含完整的仓库信息：
- 地理：country / state / city / zipCode / lat / lng
- 配置：timeZone / zipcodeRange / warehouseSort
- 渠道：channelNo / channelName
- 扩展：warehouseExtData (JSONObject)

适合支持的 skill：warehouse_allocation / eta / cost / oms_query
建议归属：共享数据能力（仓库信息查询）

### 5.4 渠道+连接器完整结构（ChannelDTO + ConnectorDTO）

ChannelDTO 包含渠道和连接器信息，ConnectorDTO 包含第三方平台连接配置。

适合支持的 skill：warehouse_allocation / shipping_rate
建议归属：共享数据能力（已有 GET /channel API）


### 5.5 History/Phase 阶段采集框架（重大发现）

OMS 框架层（linker-common）内置了一套方法级阶段采集框架：

| 组件 | 说明 |
|------|------|
| @HistoryMethod | 方法级注解，声明 businessType / phase / description |
| @HistoryKey | 关键字段注解，标记在 OrderItemDO.referenceNo 上 |
| HistoryAspect | AOP 切面，自动采集请求/响应/关键字段/执行状态 |
| PhaseEvent | 事件结构（businessType/phase/methodName/requestData/responseData/status/error/keyValue/time） |
| PhaseHandler | 事件处理器接口（持久化） |

适合支持的 skill：oms_query（关键事件时间线）/ order_analysis（阶段级异常上下文）
建议归属：共享能力层（阶段历史查询）

如果 PhaseHandler 的实现将 PhaseEvent 持久化到了可查询的存储，并且有对应的查询 API，那么这套框架可以成为 oms_query 最强大的数据源——按 orderNo 查询完整的 routing history。
